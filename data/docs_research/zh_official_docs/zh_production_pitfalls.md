# 生产级 Agent 的 10 个踩坑经验

## 1. 并发分类是按调用的，不是按工具类型

同一个 `file_write` 工具，追加模式可以并发，覆盖模式不能并发。不要假设工具的并发行为是静态的——运行时按调用参数决定。

## 2. 权限检查器有副作用

权限检查器追踪拒绝次数、转换模式、更新状态。不要当纯查询函数缓存——每次调用状态可能不同。

## 3. 大多数异步工作跳过 Pending 状态

实践中工作单元直接注册为 Running。不要构建假设每个工作从 Pending 开始的 UI 或监控。

## 4. Fork 子级不能 Fork

递归 Fork 指数级放大上下文成本——8KB × 3层 × 3子 = 72KB。子 Agent 保留 Fork 工具但递归调用时被阻止。

## 5. 上下文缓存是手动失效的

添加新数据源而不添加入口失效点 → Agent 在整个会话中看到过时数据。每个变更点必须显式清除缓存。

## 6. 记忆索引有硬上限

超过上限时新条目被静默截断——Agent 看起来"忘了"最近的纠正。定期审计记忆大小。

## 7. Hook 信任是全有或全无

一个不安全的 Hook 会禁用整个扩展系统，不只是那个 Hook。设计 Hook 时要极度谨慎。

## 8. 工具默认 Fail-Closed

新工具默认拒绝所有调用，开发者必须显式标记只读模式。忘记标记只会降低吞吐量，不会造成危险。

## 9. 回收必须先通知

终态工作单元只有在父级收到完成信号后才可 GC。通知前回收 → 父级永远读不到结果。

## 10. 协调者必须自己综合

"基于你的发现，修复它"是多 Agent 协调中最反模式的做法。协调者消化 Worker 结果成精确规格，再派发。
