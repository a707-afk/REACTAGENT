# 多 Agent 协调：Coordinator、Fork、Swarm 三种模式

## 核心反模式

> "基于你的发现，修复它"

这是多 Agent 协调中最常见的反模式——协调者把 Worker 的研究结果原封不动传给实现 Worker，不做任何综合和细化。**协调者必须自己综合，不能委派理解**。

## 三种委派模式

### 1. Coordinator 模式（最安全，最慢）

**流程**：研究 Worker → 协调者消化综合 → 实现 Worker → 验证 Worker

**上下文**：Worker 不继承协调者的上下文（零继承），每次从自包含 prompt 开始。

**何时用**：复杂多阶段任务，需要严格的质量控制和方向把控。

**实现清单**：
1. 定义阶段化工作流：研究 → 综合 → 实现 → 验证
2. 为每个 Worker 写自包含 prompt（不能写"基于之前的发现"）
3. 过滤每个 Worker 的工具集，只给它需要的
4. 决定继续 vs 新建策略：上下文重叠就继续，验证阶段必须新建

**关键一步**：研究 Worker 返回后、实现 Worker 启动前，协调者执行"综合步骤"——把研究发现消化成精确的实现规格。

### 2. Fork 模式（最快，只有一层）

**流程**：父 Agent Fork 出多个子 Agent，子级继承父级全部上下文，执行独立子任务后返回。

**约束**：
- Fork 只有单层——递归 fork 指数级放大上下文成本
- 子 Agent 的工具池里保留了 Fork 工具，但递归调用时被阻止
- 结果异步返回；父 Agent 可以继续工作

**何时用**：需要快速并行探索，且共享上下文有价值。

### 3. Swarm 模式（最灵活，最难协调）

**流程**：一组对等 Agent 共享任务列表，各自认领执行。

**约束**：
- Swarm 成员不能生成其他成员——名单扁平，防止不受控增长
- 结果异步到达，通过共享任务列表通信
- 没有任何成员有全局"协调者"角色

**何时用**：长期运行的独立工作流，任务数量和类型不可预测。

## 综合步骤——协调者的核心价值

协调者不只是一个路由器。它的价值在于"综合"：

```
研究 Worker 输出：
- "Qdrant 支持 payload filter，HNSW 索引 5ms 延迟"
- "Milvus GPU_CAGRA 索引 10x 吞吐，需 GPU 显卡"

协调者综合后的实现规格：
- 对比维度：索引方式、过滤性能、硬件需求、延迟、吞吐
- 回答结构：先特性对比，再性能数据，最后适用场景建议
- 引用源：[1] Qdrant 文档 filtering 页, [2] Milvus GPU Index 文档, [3] VectorDBBench 2026
```

不经过综合直接传给实现 Worker → Worker 生成的是碎片化的信息拼接。经过综合 → Worker 生成的是有结构的对比分析。

## 面试回答要点

被问"多 Agent 怎么协调"，关键：
- 三种模式选一种说清楚（推荐 Coordinator）
- "综合步骤"是最重要的（协调者不能只做路由器）
- 自包含 prompt 原则（不依赖隐性上下文）
- Fork 单层限制（防止上下文指数爆炸）
