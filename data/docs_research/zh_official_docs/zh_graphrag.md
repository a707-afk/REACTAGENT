# GraphRAG：知识图谱 + 向量检索的融合实践

## 传统 RAG 的盲区

传统 RAG 检索文本片段，但无法处理跨文档的实体关系推理。"这种药和我的基础病有冲突吗"需要先查药的副作用成分、再查病的禁忌清单、再交叉分析——文本片段里不会同时包含这三个信息。

## GraphRAG 架构

**第一阶段：知识图谱构建**
LLM 从文档中抽取实体和关系 → 构建图（Neo4j/NetworkX）。例如：华法林-[副作用]-[出血风险]，阿司匹林-[增风险]-[出血]。两个文本片段通过实体关系连接起来了。

**第二阶段：混合检索**
- 向量检索：找语义相关的文本片段
- 图检索：Cypher 查询找多跳实体关系
- 融合：两个结果集合并，去重

**第三阶段：多跳推理**
Agent 在图上做 2-3 跳推理：Qdrant→支持→payload filter；payload filter→ACL前置过滤→企业需求。普通 RAG 只能回答"Qdrant 有什么特性"，GraphRAG 能回答"Qdrant 的哪个特性解决你的具体需求"。

## 什么时候需要 GraphRAG

关系推理密集的场景（医疗决策、法律判例、企业内部依赖分析）才需要。简单的选型对比问题用传统 RAG 更合适——不要过度工程。

## 面试回答要点
"GraghRAG 的价值是跨文档实体推理，不是替代传统 RAG。我的系统预留了 GraphRAG 扩展点——文档入库时抽取实体关系，检索时混合向量+图查询。"
