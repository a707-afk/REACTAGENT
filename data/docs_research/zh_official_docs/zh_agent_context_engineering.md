# Agent 上下文工程：选择、写回、压缩、隔离

## 上下文是预算，不是垃圾桶

Agent 的上下文窗口不是无限的。每个 token 都有成本——无论是 API 费用还是本地推理时间。上下文工程的核心问题不是"怎么塞更多"，而是"怎么让每个 token 值得它的位置"。

四种核心操作为每个 token 的存在辩护：

## 1. 选择（Select）：三级渐进披露

不要一次加载所有内容。使用三级加载策略：

| 级别 | 内容 | 大小 | 加载时机 |
|---|---|---|---|
| L1 元数据 | 工具名/描述/技能清单 | ~500 tokens | 始终在上下文 |
| L2 指令 | 工具的完整 prompt / 技能的详细说明 | ~1-5K tokens | 激活时加载 |
| L3 资源 | 引用的文件/代码/文档 | 可变 | 按需加载 |

**缓存策略**：上下文构建器是记忆化的，但手动失效。添加新数据源而不添加对应的失效点，Agent 会在整个会话中看到过时数据。

## 2. 写回（Write Back）：上下文不是只读的

Agent 不只是读上下文，它要把信息写回持久存储：

- 自动记忆条目：用户纠正、项目规则
- 后台提取输出：会话结束时的自动总结
- 任务状态：长期运行工作的进度
- 权限规则：用户批准的权限模式

写回循环是把无状态工具调用者变成学习系统的关键。

## 3. 压缩（Compaction）：长会话的救星

参考 Claude Code 的实现——不是简单截断旧消息，而是智能摘要：

**压缩策略**：
- 保留最近 N 轮完整对话（保证当前上下文连贯）
- 旧轮次压缩为摘要（工具调用 → 输入/输出的结构化摘要）
- 标记压缩过的内容为"快照"（让模型知道需要重新获取当前状态）
- 保留关键工具调用的参数和结果（不让压缩丢失结构化数据）

**什么时候触发**：当上下文使用率超过 ~80% 时，后台异步执行压缩。

**常见失败**：简单截断（丢失工具调用上下文）→ Agent 生成幻觉参数。

## 4. 隔离（Isolate）：委派不污染父上下文

三种委派模式的上下文隔离程度：

| 模式 | 上下文共享 | 适用场景 |
|---|---|---|
| Coordinator | 无——Worker 从零开始 | 复杂多阶段任务（研究→综合→实现→验证） |
| Fork | 全部——子级继承父级历史 | 快速并行拆分，共享已加载上下文 |
| Swarm | 对等共享任务列表 | 长期运行的独立工作流 |

**关键约束**：
- Fork 只有单层——递归 Fork 指数级放大上下文成本
- Coordinator Worker 必须用"自包含 prompt"——不能写"基于之前的发现"

## 面试回答要点

- 上下文是预算（每个 token 四选一：选择/写回/压缩/隔离）
- 压缩不是截断（保留工具调用的结构化信息）
- 隔离的核心是"自包含 prompt"——不依赖调用链的隐性上下文
