# Agent 工具安全与权限管道

## 核心原则：Fail-Closed 默认

传统软件开发的默认是"能做什么"，Agent 工具的默认必须是"不能做什么"。每新增一个工具，默认拒绝所有调用，开发者必须显式打开权限。这防止了最危险的一类 bug——"我以为这个工具只是读数据，但它其实能删除"。

## 并发分类：按调用而非按类型

最容易被误解的设计决策。同一个工具对不同输入，并发安全性完全不同。

**示例**：
- `file_write("/tmp/log.txt", "append")` → 并发安全（追加模式无竞争）
- `file_write("/etc/config.json", "overwrite")` → 不允许并发（覆盖模式有竞争）
- 同一个 `file_write` 工具，两次调用的并发分类不同

**实现方式**：运行时将一批工具调用分成连续组——安全调用并行执行，任何不安全调用开始一个串行段。

## 权限管道：多源分层评估

权限不是单一的"允许/拒绝"开关。生产级 Agent 的权限评估来自多个层级，按严格优先级顺序：

1. **会话级授权**（Session Authorization）：用户在当前会话中手动批准的操作。最高优先级，会话结束即清理。
2. **CLI 参数**：启动 Agent 时传入的权限标志。
3. **命令级规则**：对特定命令（如 `rm -rf`）的硬编码拒绝规则。
4. **配置文件规则**：
   - 本地配置（项目目录 `.agent/config`）
   - 项目配置（Git 仓库根目录）
   - 用户配置（`~/.agent/config`）
   - 全局策略（组织级安全策略）
5. **工具默认值**：工具注册时声明的默认权限级别

**冲突处理**：高优先级覆盖低优先级。会话授权可以临时覆盖全局策略。

## 权限评估器的副作用

权限检查器不是纯函数，它有状态：
- 追踪拒绝次数（用于速率限制和异常检测）
- 记录转换模式（用户是否频繁覆盖某个工具的默认权限）
- 更新授权状态（批准后在同一会话内记住该决定）

**常见陷阱**：不能把权限检查器当纯查询函数缓存——每次调用可能有不同的状态。

## 工具的 Fail-Closed 默认值

工具注册时的安全默认值：

| 属性 | 默认值 | 含义 |
|---|---|---|
| `concurrent` | `false` | 默认不允许并发执行 |
| `read_only` | `false` | 默认可能修改状态 |
| `requires_approval` | `true` | 默认需要用户批准 |

开发者必须主动标记 `read_only=true` 才能跳过批准——忘记标记只读工具只会降低吞吐量，不会造成危险。

## Hook 注入点

权限不是在工具调用前一次性检查的，而是通过 Hook 在多个生命周期节点注入：

- **PreToolUse Hook**：工具调用前，可以阻止（权限不足）、修改参数（注入安全校验）、或记录日志
- **PostToolUse Hook**：工具调用后，可以审计结果、触发后续操作

**信任模型**：Hook 的信任是"全有或全无"——工作区不受信任时，所有 Hook 被禁用，不只是可疑的那几个。这简单但粗糙。

## 面试回答要点

被问"Agent 怎么安全地调工具"，关键点：
- Fail-closed 默认（不是 fail-open）
- 并发控制是按调用而非按类型（给出 file_write 的例子）
- 权限管道多层评估（会话>CLI>命令级>配置>默认）
- 权限检查器有状态（不是纯函数）
