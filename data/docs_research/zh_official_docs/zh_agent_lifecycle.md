# Agent 生命周期管理：Hook、Bootstrap、长期运行工作

## Hook：在生命周期关键时刻注入逻辑

Hook 让你在不修改核心 Agent 代码的前提下扩展行为。定义触发时刻 + 执行的副作用。

**Hook 触发时刻**：
- **PreToolUse**：工具调用前（可以阻止、修改参数、记录日志）
- **PostToolUse**：工具调用后（审计结果、触发后续操作）
- **PrePromptSubmit**：prompt 提交到模型前（修改上下文）
- **AgentStart / AgentStop**：Agent 启动/停止时（初始化/清理资源）

**信任模型**：Hook 的信任是"全有或全无"。工作区不受信任时，所有 Hook 被禁用——不只是可疑的那几个。这简单但粗糙：一个不安全的 Hook 会禁用整个扩展系统。

**会话级 Hook**：临时 Hook，会话结束时自动清理。适合单次调试或一次性的审计需求。

## Bootstrap：按依赖顺序启动

Agent 启动不是简单的"加载所有模块"。Bootstrap 是按依赖拓扑序的阶段化初始化：

1. **基础配置**：加载环境变量、CLI 参数
2. **安全子系统**：初始化权限管道、Hook 引擎
3. **数据层**：连接数据库、加载向量库、预热缓存
4. **工具注册**：注册工具、验证 schema、加载技能
5. **网络监听**：启动 HTTP/gRPC 服务

**信任边界**：用户授权同意的时刻是关键拐点。安全敏感的子系统（遥测、密钥加载）不能在信任建立前激活。

**记忆化**：每个 Bootstrap 阶段的产物被缓存。不重复执行已完成的初始化。

## 长期运行工作：状态机追踪

对于需要数分钟到数小时的异步工作（如研究任务、reindex），不能用简单的"运行中/完成"布尔值追踪。需要完整的状态机：

```
Pending → Running → Completed
                 → Failed (可重试)
                 → Killed (不可重试)
```

**回收协议**（最容易出错的部分）：
1. 工作单元进入终态（Completed/Failed/Killed）
2. 立即清理磁盘备份（释放空间）
3. 只有在**父级收到完成信号后**才从内存中移除
4. 在通知前回收 → 产生竞态 → 父级永远读不到结果

**常见陷阱**：大多数异步工作跳过 "Pending" 状态——实际注册时直接标记为 "Running"。不要构建假设每个工作从 Pending 开始的 UI。

## 面试回答要点

- Hook 是注入点，不是继承层次（全有或全无信任）
- Bootstrap 是按依赖拓扑序的阶段化初始化（记忆化 + 信任边界）
- 长期运行工作的回收协议（通知前不能 GC）
