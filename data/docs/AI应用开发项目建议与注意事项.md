# AI 应用开发项目建议与注意事项

> 适用目标：以 AI 应用开发岗位为导向，通过项目把 RAG、Agent、后端工程、评估和面试表达整合起来。

---

## 一、当前阶段判断

你现在不是“没学会”，而是处在：

> 概念已经有框架，但还没有通过项目形成主动输出能力。

典型表现：

- 看答案能理解，但闭卷回答不完整。
- 知识点在脑子里是散的，没有形成工程链路。
- 知道 RAG、Agent、Tool Use、Memory、Planning 等概念，但还不能稳定讲清“为什么这么设计、出了问题怎么排查”。

接下来不建议继续泛泛看讲义，而是进入项目阶段。

核心目标不是“学完所有知识”，而是：

> 做出一个能运行、能评估、能讲清楚、能解释边界的 AI 应用项目。

---

## 二、项目路线建议

建议先做 RAG 项目，再做 Agent 项目。

不要一开始就做复杂多 Agent 系统。Agent 项目如果没有稳定的检索、工具、评估基础，很容易变成看起来很复杂但不可控的 Demo。

### 项目 1：企业知识库 RAG 问答系统

可以先用自己的学习 Markdown 文档作为数据源，后续再换成企业制度、产品手册、合同、招投标文档、客服知识库等。

最小闭环：

```text
文档读取
→ 文档清洗
→ 分块
→ Embedding
→ 向量库入库
→ 用户提问
→ 检索 Top-K
→ LLM 生成回答
→ 返回引用来源
```

逐步迭代：

| 阶段 | 目标 | 产出 |
|---|---|---|
| V1 | 跑通基础 RAG | 能问答，能返回来源 |
| V2 | 增加评估集 | 20-50 条测试问题 |
| V3 | 混合检索 | BM25 + 向量检索 |
| V4 | Reranker | 粗召回后精排 |
| V5 | Query Rewrite | 改善口语化、模糊问题 |
| V6 | 引用校验 | 回答必须有来源 |
| V7 | FastAPI + 前端 | 可交互 Demo |
| V8 | Docker + README | 可部署、可展示 |

### 项目 2：学习助手 / 研究助手 Agent

在 RAG 项目稳定后，再做 Agent。

最小 Agent 版本：

```text
用户输入复杂问题
→ Agent 判断是否需要检索
→ 拆解子问题
→ 调用知识库检索工具
→ 汇总生成回答
→ 自检引用与结论
→ 输出结构化结果
```

建议工具：

- `search_kb`：检索知识库。
- `verify_citations`：校验回答是否有依据。
- `summarize_context`：压缩检索结果。

不要一开始就加入太多功能，例如复杂长期记忆、多 Agent 辩论、MCP Server、自动执行外部高风险操作等。先把状态流、工具调用、错误处理做扎实。

## 三、AI Coding 协作方式

现在项目基本都会用 AI Coding，这不是问题。关键是你不能把 AI 当“代写”，而要把它当“初级工程师”。

你的角色是：

```text
产品负责人
架构负责人
验收负责人
调试负责人
面试讲解者
```

AI 的角色是：

```text
代码生成助手
重构助手
测试补全助手
文档助手
排查建议助手
```

### 你必须亲自负责的部分

这些不能完全交给 AI：

1. 需求定义

你要能说清：

```text
这个系统给谁用？
解决什么问题？
输入是什么？
输出是什么？
什么情况算成功？
什么情况必须拒答？
什么情况需要人工确认？
```

2. 架构决策

你要能解释：

```text
为什么用 FastAPI？
为什么用这个向量库？
为什么要分块？
为什么要 Reranker？
为什么需要 Query Rewrite？
为什么要做评估集？
```

3. 核心链路理解

你必须能闭卷画出：

```text
文档加载 → 清洗 → 分块 → Embedding → 向量库
用户问题 → Query Rewrite → 检索 → Rerank → Prompt 拼接 → LLM 生成 → 引用校验 → 输出
```

4. 调试和验收

你要会判断：

```text
检索结果是否相关？
回答有没有依据？
引用是否真实？
检索不到时是否拒答？
日志能不能定位问题？
延迟是否可接受？
成本是否可控？
```

5. 面试讲解

代码可以由 AI 辅助生成，但项目讲解必须由你完成。

你要能说清：

```text
我一开始怎么做？
遇到什么问题？
怎么定位？
怎么改？
改完效果如何？
还有哪些限制？
```

### 可以交给 AI Coding 的部分

这些可以大量让 AI 辅助：

- FastAPI 项目骨架。
- Pydantic Schema。
- 文档读取模块。
- 分块函数。
- 向量库封装。
- Embedding 调用封装。
- Reranker 调用封装。
- 前端页面。
- Dockerfile。
- Docker Compose。
- README 初稿。
- 单元测试初稿。
- 日志和配置模块初稿。

但 AI 写完后，你必须读懂核心文件，至少能解释每个模块的职责。

## 四、给 AI Coding 的正确任务拆分

不要一次性说：

```text
帮我做一个完整 RAG 系统。
```

这会导致代码失控、结构混乱、你也学不到东西。

应该按小任务推进：

```text
第 1 步：搭 FastAPI 项目骨架，包括配置、日志、健康检查接口。
第 2 步：实现 Markdown 文档加载和分块模块，并写脚本验证输出。
第 3 步：接入 Embedding，把 chunk 写入向量库。
第 4 步：实现 query 接口，只返回 Top-K chunk，暂时不调用 LLM。
第 5 步：基于检索结果调用 LLM 生成带引用回答。
第 6 步：加入评估脚本，批量运行 eval_questions.jsonl。
第 7 步：加入 BM25 + 向量混合检索。
第 8 步：加入 Reranker。
第 9 步：加入 Query Rewrite。
第 10 步：补充前端、Docker、README。
```

每一步都必须能运行、能验收，再进入下一步。

## 五、提示词模板

每次让 AI Coding 写代码时，建议包含 6 个要素：

```text
1. 当前目标
2. 已有文件结构
3. 输入输出格式
4. 技术约束
5. 验收标准
6. 不要做什么
```

示例：

```text
请实现 Markdown 文档加载与分块模块。

目标：
读取 data/docs 目录下所有 .md 文件，按标题和段落进行分块。

要求：
- 每个 chunk 包含 content、source、chunk_id、heading、char_count
- chunk 长度控制在 500-800 中文字符
- 保留来源文件名和标题路径
- 不要在这个步骤接入 embedding
- 不要引入复杂框架

验收：
- 写 scripts/test_chunking.py
- 运行后打印前 5 个 chunk
- 能看到 chunk 内容、来源文件、标题、字符数
```

## 六、每天必须沉淀的内容

每天不要只写代码。必须留下项目记录。

建议每天记录：

```text
1. 今天完成了什么？
2. 遇到什么问题？
3. 我怎么定位？
4. 最后怎么解决？
5. 这个问题面试时可以怎么讲？
```

示例：

```text
今天完成了基础向量检索接口。
问题：用户问“这个怎么用”时，检索结果不稳定。
定位：原始 query 太口语化，和文档标题相似度低。
解决：后续计划加入 Query Rewrite，把口语问题改写成更适合检索的表达。
面试表达：用户查询质量会直接影响召回，所以我在检索前增加了 Query Rewrite。
```

这类记录会直接变成你的面试素材。

## 七、项目验收标准

一个能支撑 20-25K AI 应用开发面试的项目，至少应该具备：

- 有明确业务场景。
- 有真实或接近真实的数据。
- 能本地运行。
- 有 API。
- 有简单前端或可交互界面。
- 有 Docker Compose。
- 有 README。
- 有架构图。
- 有 20-50 条评估问题。
- 有优化前后对比。
- 有 bad case 分析。
- 有 AI 能力边界说明。
- 有失败处理和拒答策略。

不要只展示“能问答”。要展示你如何让它从不稳定变得更可靠。

## 八、AI 能力边界

AI 应用开发必须知道哪些能做，哪些不能承诺。

### 适合 AI 做的事

- 文档问答。
- 摘要生成。
- 多文档对比。
- 信息抽取。
- 文本分类。
- 初稿生成。
- 辅助检索。
- 半自动流程推进。
- 规则明确场景下的工具调用。

### 不应直接承诺的事

- 100% 正确。
- 完全替代专家判断。
- 自动处理高风险法律、医疗、财务决策。
- 无人工审批地付款、删库、发正式合同。
- 没有数据也能准确回答专业问题。
- 没有权限边界也能安全接入所有系统。

面对客户幻想时，可以这样表达：

```text
这个可以做成辅助决策系统，但不建议让 AI 直接自动决策。
我们可以让它先检索资料、生成建议、标注依据，最后由人工确认。
```

## 九、面试表达训练

项目开发过程中，每天练 2-3 个闭卷问题。

示例问题：

- RAG 为什么会幻觉？
- Chunk Size 怎么定？
- Reranker 和 Embedding 检索有什么区别？
- BM25 和向量检索为什么要结合？
- Query Rewrite 解决什么问题？
- 检索不到怎么办？
- 如何降低 AI 胡编？
- Agent 和普通工作流有什么区别？
- Function Calling 和 MCP 是什么关系？
- 客户要求 AI 自动审批合同，你怎么设计？

回答结构：

```text
是什么？
为什么需要？
工程上怎么做？
会出什么问题？
我在项目里怎么处理？
```

示例：

```text
Reranker 是粗召回之后的精排模块。
向量检索负责从大量文档里快速召回候选 chunk，但排序不一定最准确。
所以我会先用向量检索或 BM25+向量召回 Top 20-50，再用 Reranker 选 Top 3-8 给 LLM。
问题是延迟和成本会上升，所以需要控制候选数量、做缓存，并记录每次检索和精排耗时。
在我的项目里，我会对比加 Reranker 前后的命中率和回答质量。
```

## 十、其他技术学习优先级

当前不要发散去深学 Kafka、Kubernetes、复杂微服务治理。

优先级如下。

### 必须扎实

- Python
- FastAPI
- Pydantic
- SQL
- MySQL 或 PostgreSQL
- Redis 基础缓存
- Docker / Docker Compose
- Git
- Linux 基础命令
- HTTP / REST API
- 日志与异常处理
- 配置管理
- 简单鉴权

### AI 应用重点

- RAG
- Embedding
- Chunking
- 向量库
- BM25
- Reranker
- Query Rewrite
- RAG 评估
- Tool Calling
- LangGraph
- Agent 状态管理
- AI 安全边界
- 成本和延迟控制

### 可以暂缓

- Kafka
- Kubernetes
- 高并发深水区
- 复杂 MLOps
- 大模型底层训练
- 分布式系统细节

Kafka 可以先了解用途：

```text
消息队列，用于异步解耦、削峰填谷、日志流处理。
```

但它不是当前 AI 应用开发求职的第一短板。

## 十一、推荐执行节奏

未来 4-6 周建议按下面推进：

```text
第 1 周：RAG 最小闭环，完成文档入库、检索、问答、引用。
第 2 周：加入评估集，记录 bad case，完善日志。
第 3 周：加入 BM25、Reranker、Query Rewrite。
第 4 周：FastAPI、前端、Docker、README、架构图。
第 5 周：做 LangGraph Agent 小项目。
第 6 周：项目复盘、面试表达、投递简历。
```

每天建议时间分配：

```text
4 小时：写项目代码和调试
2 小时：补项目卡点相关知识
1 小时：整理项目记录
1 小时：闭卷面试表达
```

---

## 十二、最终判断标准

不要用“我看了多少教程”判断学习效果。

用下面几个问题判断：

- 我能不能从零讲清一个 RAG 系统的完整链路？
- 我能不能解释每个模块为什么存在？
- 我能不能说出项目里的 3 个 bad case？
- 我能不能说明我如何优化过检索质量？
- 我能不能说明 AI 哪些地方不能直接自动化？
- 我能不能把项目本地跑起来，并让别人试用？
- 我能不能用 3 分钟讲清项目，用 10 分钟讲深项目？

如果这些能做到，你就不是只“学过 AI 应用开发”，而是已经开始具备 AI 应用开发工程师的项目表达能力。

---

## 十三、一句话原则

> 代码可以由 AI 辅助生成，但需求、架构、验收、调试、边界判断和项目讲解必须由你主导。
