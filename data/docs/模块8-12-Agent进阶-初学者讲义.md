# 模块 8：Agent 规划（Planning）

## 8.1 初学者最容易混淆的事

- **「规划」不是「检索」**：检索是「去知识库里拿资料」；规划是「先决定**做哪几步、顺序怎样**」，里面**可以包含**「第 3 步去检索」，但二者不是同一件事。  
- **「规划」也不是「写提示词技巧」**：提示词只管这一轮怎么说；规划管**多轮、多步、可中途调整**的整体策略。

## 8.2 用生活类比理解

你要组织一次小型活动：

- **没有规划**：想到啥干啥，容易漏订场地、忘通知人。  
- **有规划**：目标 → 拆成「预算 / 场地 / 人员 / 物料」→ 每步有人负责 → 如果场地没订上 → **改计划**（换日期或换场地），而不是假装活动照常进行。

Agent 做复杂问题（写调研报告、多步排障、多工具协作）同理：**先有计划，再执行；执行中发现不对，再更新计划。**

## 8.3 核心概念逐个讲

### ① 任务分解（Task Decomposition）

**含义**：把一个大目标拆成**子任务序列**（有时是树状，有时是 DAG——有依赖关系）。

**例子（写「公司产品介绍」）**：

1. 确认产品名称与版本  
2. 检索内部文档中与功能相关的段落  
3. 检索公开资料中与竞品对比有关的信息  
4. 按模板草拟一稿  
5. 自检事实是否与文档一致  

每一步都可以对应图里的一个节点、或 Crew 里的一个 Task。

**初学者注意**：子任务要 **MECE 一点**（不严重重叠、尽量不漏大块）；实际系统里常由 **LLM 生成草稿计划**，再由**规则/人**裁剪。

### ② 三种常见规划策略（背名字 + 能举例）

| 策略 | 白话 | 适用直觉 |
|------|------|----------|
| **自顶向下** | 从大目标一路拆成小目标 | 写报告、做方案、结构化交付物 |
| **自底向上** | 手里已有不少碎片结果，再「拼」成答案 | 工具多、中间结果杂、最后要汇总 |
| **LLM 生成计划** | 让模型直接输出步骤列表（或 JSON） | 灵活，但要校验步骤是否可行、是否会幻觉 |

### ③ 执行监控（Execution Monitoring）

**要监视什么**（初学者按这个清单想即可）：

- **进度**：当前第几步？是否完成？  
- **成败**：工具报错了吗？检索结果为空吗？  
- **偏离**：用户中途改需求了吗？上一步输出是否让下一步无法继续？

**动态重规划（Re-planning）**：

- 不是「失败了程序就退出」，而是：**根据新状态再生成一份计划**（或只改后面几步）。  
- 这和 **Adaptive RAG**（简单问不检索、复杂问多检索）是同一类思想：**策略随情况变**。

## 8.4 和 RAG 怎么绑在一起（面试常问）

规划层要显式回答两件事：

1. **何时检索**：一开始就检索？还是先澄清问题再检索？  
2. **检索什么**：整句 query 检索？还是拆成子问题分别检索？

**差的流程**：每一步都疯狂检索，噪声大、贵、慢。  
**好的流程**：计划里写清——「先要实体名，再检索该实体的规格表」——这与你在模块 2 里学的表格/结构化检索顺序也能呼应（先语义找行，再结构化查列）。

## 8.5 和你正在练的 LangGraph 的关系

- **State** 里可以加字段：`plan: list[str]`、`current_step: int`、`last_error: str`。  
- **节点**：`plan_node`（生成/更新计划）→ `execute_step_node` → `verify_node` → 条件边决定是否**回到 plan**。  
- 不必一上来就写很复杂：**先手写固定计划**，再换成 LLM 生成计划。

## 8.6 小练习（纸上就能做）

1. 选一个任务：「给用户写一份 300 字的入职指引」。手写 5 步计划。  
2. 标出：哪一步**必须**检索知识库？哪一步**不能**检索（只能靠常识或模板）？  
3. 假设第 2 步检索结果为空，你会**怎么改计划**？写两行说明。

# 模块 9：Agent 记忆（Memory）

## 9.1 为什么需要单独学「记忆」

大模型**本身**不会自动记住你三个月前说过的话 —— 除非你每次把历史**塞进当前请求**，或**写进外部存储**再在合适时候**读回来**。

Agent 要「像人协作一样」长期跟进任务，就必须有**记忆系统的设计**，否则：

- 要么 **忘光**（每次像新用户）；  
- 要么 **全文塞上下文**（贵、超 token、噪声大）；  
- 要么 **乱检索旧对话**（记忆污染，答非所问）。

## 9.2 三层记忆（大纲版）—— 初学者按这张表记就够

### ① 短期记忆（Short-term）

**是什么**：**当前这一轮对话/当前任务**里，模型能直接看到的消息（受窗口长度限制）。

**怎么实现**：  
- 最常见：`messages` 列表；LangGraph 里用 `MessagesState` + `add_messages` **自动合并**新消息。  
- **截断**：只保留最近 N 轮或 N token（要丢细节时，配合摘要）。

**典型错误**：以为「短期记忆」= 向量库。不是。**向量库一般是长期记忆的落库方式之一。**

### ② 长期记忆（Long-term）

**是什么**：**跨会话**仍希望保留的东西：用户偏好、项目背景、上周的结论、常用术语表等。

**怎么实现（概念级别）**：

- **向量检索**：把历史发言或摘要 embedding，按相似度取出来。  
- **摘要存储**：定期把对话压成一段「用户档案」文本，存数据库。  
- **结构化档案**：JSON/表结构存「用户 ID → 偏好」，查询时不用语义检索。

**和 RAG 知识库的区别**（面试必会）：

| | 用户/会话记忆 | 企业 RAG 知识库 |
|---|----------------|------------------|
| 内容 | 你是谁、你爱怎么用 | 制度、产品说明、内部文档 |
| 权限 | 常按用户隔离 | 常按部门/密级隔离 |
| 更新频率 | 高（对话产生） | 中（文档发布节奏） |

两者**技术可以类似**，**产品语义不要混**。

### ③ 工作记忆（Working Memory / Scratchpad）

**是什么**：**当前这个任务**的「草稿纸」：子问题列表、中间计算、todo、上一轮工具返回的精简版。

**放哪**：  
- 图的 **state** 里单独字段：`scratchpad: str` 或 `notes: list`。  
- 不要让所有中间结果都进 `messages`（会爆炸），**该压缩就压缩**。

## 9.3 记忆检索：何时查、怎么避免「记忆污染」

**何时检索长期记忆**（简单规则即可入门）：

- 用户说「上次」「之前」「按我喜欢的风格」→ 倾向查。  
- 用户问纯事实、且知识库已有权威文档 → **先查知识库**，不必先翻个人闲聊记忆。

**记忆污染**指：把**不相关**的旧对话片段检索进来，模型会 **强行用旧信息回答新问题**。

**常见缓解**：

- 相似度 **阈值**（不够像就不采纳）。  
- **话题/会话 ID** 隔离（不要跨用户混记忆）。  
- **摘要层级**（用摘要代替全文检索）。  
- 定期 **清理** 错误或过期记忆（产品上要配「重置记忆」）。

## 9.4 MemGPT 与「记忆压缩」（进阶但面试可能提）

**MemGPT（概念）**：把「哪些留在窗口里、哪些写到外存」当成一种**管理机制**来优化——像操作系统的内存与外存交换。初学者不必抠论文公式，记住**目的**即可：**在有限上下文里尽量保持有用信息**。

**记忆压缩**：长对话 → **摘要**再入库；或多级摘要（日摘要、周摘要）。和 RAG 里「上下文压缩」是近亲。

## 9.5 小练习

1. 设计三张表（纸上）：短期 / 长期 / 工作记忆 **各存什么字段**。  
2. 举一个「应该查长期记忆」的问句、一个「不该查长期记忆」的问句。  
3. 想一个「记忆污染」场景：例如把 A 用户的喜好混到 B 用户——怎么用工程手段避免？

# 模块 10：多 Agent 协作 + A2A 协议

## 10.1 为什么需要「多个 Agent」

一个 Agent **既当大脑又当所有手**时常见问题：

- **提示词超长**（角色太多）。  
- **工具太多**（模型选错工具概率升）。  
- **权限难做**（同一个 Agent 又有读库又有删库能力很可怕）。

**拆开**后：每个 Agent **角色窄、权限清晰**，再由上层负责协作。

## 10.2 三种架构模式（用协作关系记）

### ① Orchestrator–Worker（总管 + 工人）

- **谁指挥**：一个 Orchestrator（调度者）。  
- **谁干活**：多个 Worker（专员）。  
- **像**：项目经理分配工单。  
- **优点**：好控制、好调试。  
- **缺点**：总管可能成瓶颈；总管规划错了全军覆没（所以要监控与重规划，见模块 8）。

### ② Peer-to-Peer（对等）

- **谁指挥**：没有唯一中心，Agent 互相商量。  
- **像**：小组脑暴。  
- **优点**：灵活。  
- **缺点**：难收敛，要设计**停止条件**和**仲裁**，否则容易循环扯皮。

### ③ Hierarchical（层级）

- **多层**：高层拆大任务，中层拆部门任务，底层执行。  
- **像**：大公司组织架构。  
- **适合**：强流程、强合规、任务天然分层。

## 10.3 协作机制要解决的工程问题

**消息传递**：Agent A 把**什么格式**的东西发给 B？（自然语言摘要 / JSON 任务单 / 仅 task id）

**共享状态**：

- **黑板模式**：大家读写同一块 state（要想好锁与版本）。  
- **事件流**：Kafka 等（企业里常见，初学者知道「有大图这个方向」即可）。

**冲突解决**：

- 两个 Agent 结论打架怎么办？规则：优先级、第三裁判 Agent、**人审**（接模块 12 的安全与 HITL）。

## 10.4 和大纲里两种框架的对应

- **CrewAI**：**角色 + 任务 + Crew** 很贴「总管–工人」叙事，做 demo 快。  
- **LangGraph**：**子图**把子团队封装成节点；**状态机**特别适合「审批、分支、回放」（你已在阶段练习里接触）。

**不要被名字吓到**：「多 Agent」不一定是多个 LLM 实例——有时是**同一模型、多个 system prompt 角色的轮流调用**；工程上关心的是**边界与权限**。

## 10.5 A2A 协议（Agent-to-Agent）— 初学者版

**问题**：协作不只发生在**你一个仓库里**。别的团队用别的框架、语言、部署方式，你怎么叫他们「帮你干一段活」？

**A2A 的定位**：  
像给 Agent 之间规定一套**外交公文格式**：**怎么发现对方能干什么、怎么发起任务、怎么拿结果**。你的规划里有一句必背：

- **MCP**：Agent **连接工具和数据**（垂直集成）。  
- **A2A**：Agent **连接另一个 Agent**（水平协作）。  

**二者不互斥**：常见架构是「用 MCP 接数据库，用 A2A 接外部门的审核 Agent」。

**你暂时不用背协议字段**：入门先记 **Agent Card（代理名片）**、**Task（任务）**、**消息 / 工件（交付物）** 三类对象，知道「是干什么用的」即可；细节以官方规范为准。

## 10.6 小练习

1. 画三个框：Orchestrator、Worker-A、Worker-B，箭头上标「派发任务 / 返回结果」。  
2. 写一句：你的项目里 **哪种情况适合 Hierarchical**（不用对，能自圆其说即可）。  
3. 用中文各写一句：**MCP 解决什么**、**A2A 解决什么**。

# 模块 11：Agent 框架深入（在 6～7 基础上的「加深」）

> 模块 6～7 让你**认识** LangGraph、CrewAI、OpenAI Agents SDK、Smolagents、Google ADK，并会简单选型。  
> 模块 11 是：**选定之后，生产里要抠哪些点**。

## 11.1 LangGraph（深入清单）

| 主题 | 你要能说出什么 |
|------|----------------|
| **子图** | 把一整块逻辑封装成一个「超级节点」，大图更清晰；子团队可独立测试。 |
| **状态持久化** | checkpoint + `thread_id`：崩溃可恢复、同一用户多会话可区分。 |
| **Human-in-the-loop** | `interrupt_before`、恢复执行、`update_state` 干预；和「高风险操作」绑定。 |

**初学者实践路径**：你已有 `LangGraph学习路线.md` 的阶段 1～5；深入时优先把 **checkpoint + HITL** 做稳，再学子图。

## 11.2 CrewAI（深入清单）

- **角色**：`role / goal / backstory` 怎样写才不「玄学」？—— 约束输出格式、约束工具使用边界。  
- **任务流**：多个 Task 的 `context`、顺序与层级（`hierarchical`）何时用。  
- **工具分配**：哪个 Agent 能调用删库？—— **最小权限**。

## 11.3 OpenPII Agents SDK（深入清单）

- **handoff**：控制权怎么交接、上下文带哪些、如何防止交接丢关键信息。  
- **guardrails**：哪些输入/输出必须拦截。  
- **tracing**：出问题能否复现「当时走了哪几个 Agent」。

## 11.4 Smolagents（深入清单）

- **CodeAgent**：写代码执行——灵活，但**安全沙箱**是生死线。  
- **ToolAgent**：走工具调用——更可控。  
- 初学者结论：**教学玩 CodeAgent 很爽，生产要默认假设「代码有恶意」**。

## 11.5 Google ADK（深入清单）

- **A2A**：把 ADK 做的 Agent 暴露成可被外部调用的协作节点（概念 + 官方示例路线）。  
- **Gemini 多模态**：图/音/视频进线时和纯文本 Agent 的差异。  
- **GCP 部署**：和你是否已上云强相关；没上云可先记「生态位」。

## 11.6 选型怎么答才像「有经验」（模板）

**不要说**：「因为 LangGraph 热门。」  
**要说**（举例结构）：

1. **约束**：要不要人在回路？要不要强审计与回放？  
2. **团队**：更熟 Python 状态机还是更熟「角色剧本」？  
3. **生态**：是否绑 OpenAI / GCP / 本地 HF？  
4. **风险**：有没有代码执行？有没有写库工具？

# 模块 12：Agent 评估与安全

## 12.1 为什么要单独学「评估」

Agent = LLM + 工具 + 规划 + 记忆，**出错维度比单次问答多得多**：

- 计划根本不该执行那一步  
- 工具选错、参数错  
- 检索没检索、或检索了但不引用  
- 泄露了不该泄露的数据

所以指标不能只用一个「像不像人」。

## 12.2 指标怎么记（大纲四项）

| 指标 | 含义（白话） |
|------|----------------|
| **任务完成率** | 用户要的事做完没有 |
| **工具使用正确率** | 该不该用工具、用哪个、参数对不对 |
| **规划合理性** | 步骤是否绕远、是否多余、是否可解释 |
| **响应时间** | 迟了用户就走；也和成本相关 |

**和模块 4 的衔接**：模块 4 的 Faithfulness / Relevancy 等偏 **RAG 回答质量**；模块 12 更偏 **Agent 行为与系统风险**。两边合在一起才是「整条链路」。

## 12.3 三种评估手段

1. **人工评测**：准，贵；适合做金标、做 rubric。  
2. **LLM-as-Judge**：快，适合回归；注意评委模型偏差。  
3. **自动化测试**：给固定场景，断言「必须调用 search」「禁止调用 delete」等。

**初学者建议**：从 **10 条固定测试用例** 开始写，比空谈指标有用。

## 12.4 安全威胁要分两层（v2 很重要）

### ① LLM / Agent 应用层（传统安全题）

- **Prompt Injection**：恶意输入诱导模型违背策略。  
- **工具滥用**：模型或攻击者诱导调用危险工具（删数据、发邮件）。  
- **数据泄露**：把内部文档、PII 打进外部模型或日志。  
- **权限越界**：A 用户数据被 B 用户问出来（经常和记忆/RAG 隔离失败一起出现）。

### ② 知识库层（和模块 5.4 呼应，别只会 Prompt Injection）

- **BadRAG / 投毒**：库里被塞了假文档，模型「忠实」地引用假东西。  
- **TrojanRAG 类触发**：平时正常，遇到特定触发词才作恶。

**面试答题姿势**：  
「RAG/Agent 安全至少两层：**模型输入输出**一层，**知识库数据面**一层。」

## 12.5 防护策略（背框架，再举一例）

- **输入过滤**：可疑模式、越权指令。  
- **工具权限**：白名单、只读账号、审批（接 HITL）。  
- **输出审查**：PII、违规话术、幻觉敏感域二次校验。  
- **操作审计**：谁、何时、调了什么工具、改了什么数据。  
- **MCP 边界**：连哪个 Server、是否可信版本、是否可写。
