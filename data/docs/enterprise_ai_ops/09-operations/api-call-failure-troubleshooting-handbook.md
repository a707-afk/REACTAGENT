# API 调用失败排查手册

## 适用范围

适用于客户或内部系统调用平台公开 API（创建工单、查询状态、Webhook 投递等）出现错误码、超时、重试失效等问题的一线排查与升级分界。不涉及具体密钥内容与生产配置截图的外发。

## 处理流程

1. **固化现象**：HTTP 状态、业务错误码、请求 ID（如有）、近似时间戳、受影响环境（沙箱/生产）。  
2. **本地复现**：用最小示例重放；校验时钟偏差、DNS、TLS 版本、出站代理。  
3. **服务端核查**：配额、限流、IP 准入、签名算法、时钟窗口、幂等键冲突记录。  
4. **链路检查**：网关、上游依赖、熔断状态；判断是否区域性故障。  
5. **客户沟通**：提供可公开的缓解建议（退避重试节奏、切换到只读接口等）。  
6. **事后项**：若为缺陷或文档不清晰，开立改进单并回填知识条目。

## AI 可以做什么

- 将杂乱日志条目整理为标准排查提纲。  
- 对照手册解释常见错误码含义与下一步检查项。  
- 起草面向客户的技术邮件模板（仍需工程复核）。

## 常见误区与纠偏

误区包括未保存请求追踪 ID 就与研发反复对齐，或是把间歇性超时全部归结为客户端抖动。另一类是在限流场景中建议“无限加大重试频率”，反而会触发更严格封禁。跨环境混用密钥、端点与版本号也会造成难以复现的假阳性，应先冻结变量再缩小范围。

## 度量与复盘

按月汇总错误码分布、区域性差异、以及与发布窗口的相关性；对每条严重故障回填“文档是否已覆盖该类错误”的结论。对客户侧常见问题制作自助诊断清单以减少重复工单。对大客户可约定共享只读仪表盘而非实时全量日志外发。

## AI 禁止做什么/必须人工确认

- AI 不得指导客户绕过安全策略或禁用审计。  
- 不得凭空推断根因在未验证前告知客户“已确认为某组件故障”。  
- 不得粘贴真实 Token、密钥、完整请求体中的敏感载荷。

## 推荐回复或面试表达要点

- “API 排查先看**可追溯 ID**，再看**配额与签名**，最后才怀疑平台。”  
- “我们把**最小复现**放在第一步，节省时间。”  
- “对客户的结论要经得起**服务端日志对齐**。”
