# 项目二执行手册（给你自己看的）

> 配套文档：`PROJECT-2-MULTI-AGENT.md`（那份是"做什么"，这份是"你该怎么想、什么时候做什么"）
> 特别提醒：你说过 Agent 不懂。这份手册会比项目一更强调"先理解再动手"，每个阶段先讲清概念你才不会变成纯调包。

---

## 写在前面：做 Agent 项目最容易犯的错

1. **直接上多 Agent**：还没搞懂单个 ReAct 怎么转，就堆四个 Agent → 一团乱，面试讲不清
2. **工具一把梭**：定义一个万能工具，LLM 根本不知道啥时候用 → 工具选择准确率崩
3. **不做记忆压缩**：Agent 跑几步上下文就爆 → token 爆炸，长任务失败
4. **没有消融实验**：四个 Agent 堆一起，但说不出每个组件贡献多少 → 面试官问"多 Agent 比单 Agent 好多少"答不出

**这份手册的顺序是精心设计的：单 Agent → 记忆 → 多 Agent → 反思。一层稳了再加下一层，每层都有数据。**

## 先决条件：动手前必须搞懂的 4 个概念

⚠️ 在写任何代码前，先确认你能脱稿讲清这 4 个（看项目方案第一部分 + 知识地图的 Agent 章节）：

1. **Agent 和 Pipeline 的区别** = 谁决定下一步（LLM 决策 vs 写死路径）
2. **ReAct 循环** = Thought→Action→Observation 循环，能默写一个例子
3. **四大组件** = Planning / Memory / Tools / Action 各是什么
4. **为什么要多 Agent** = 上下文隔离 + 能力专精 + 可并行

搞不懂这 4 个就动手，你做出来的还是个你讲不清的玩具。**这一步花一天看懂，比急着写代码值。**

## 工程化能力在 Agent 项目里怎么体现？

Agent 项目的工程化信号和 RAG 项目不同，更偏"系统设计"和"鲁棒性"：

| 工程化信号 | 在哪体现 | 面试官想看到 |
|-----------|---------|-------------|
| 工具系统设计 | 阶段0 | 工具有清晰边界、Schema 规范、权限/超时/重试 |
| 状态机设计 | 阶段1、3 | 你能驾驭复杂的状态流转，不是面条代码 |
| 鲁棒性/容错 | 阶段1 | Agent 失控、工具失败时不崩，有降级 |
| 上下文工程 | 阶段2 | 记忆分层压缩，token 可控（核心能力） |
| 并发/调度 | 阶段3 | DAG 调度、并行执行、依赖管理 |
| 可观测性 | 阶段5 | 能回放多 Agent 决策树，定位哪个 Agent 出错 |
| 安全隔离 | 阶段0 | 代码执行沙箱隔离 |
| 成本意识 | 贯穿 | 分级用模型、记忆压缩、缓存 |

**Agent 项目最值钱的工程化能力是"上下文工程"（记忆压缩）和"鲁棒性"（容错降级）。这两个做好，面试就赢了一半。**

## 阶段0：工具层 + 沙箱（约 20h）

### 这个阶段的本质
工具是 Agent 的手脚。工具设计不好，后面 Agent 再聪明也白搭。先把工具做扎实。

### 必须停下来思考的决策点

**决策点 1：工具粒度怎么切？**
- ❗核心：工具要有清晰边界。`search_statute` / `search_case` 分开，不要一个 `search_everything`
- 思考：为什么分开？因为 LLM 靠 description 选工具，边界清晰它才选得准
- 这直接关系到"工具选择准确率"这个指标

**决策点 2：工具描述（description）怎么写？**
- ❗这是 Agent 项目最被低估的点。description 质量 = 工具选择准确率
- 亲手做：写完工具后，故意问几个边界模糊的问题，看 LLM 选得对不对，反复改 description

**决策点 3：代码执行工具怎么保证安全？**
- 思考：LLM 生成的代码直接跑在你机器上 = 灾难。必须 Docker 沙箱隔离
- 这是安全意识的体现，面试加分

### 必须亲手做的事
1. **亲手为每个工具写 JSON Schema**：参数、类型、描述。理解 Function Calling 的本质
2. **亲手测工具选择**：给 LLM 几个 query，看它选的工具对不对，调 description
3. **亲手验证沙箱**：写个恶意代码（如删文件），确认沙箱挡住了

### 要记录什么
- D-01：工具粒度设计 + 为什么这么切
- D-02：工具描述的迭代过程（选择准确率怎么提上去的）
- D-03：沙箱隔离方案

### 自检
- 工具为什么要分这么细？
- 工具描述为什么重要？怎么提升工具选择准确率？
- 代码执行怎么保证安全？

## 阶段1：单 ReAct Agent（约 25h）

### 这个阶段的本质
❗**这是整个项目的地基。单个 ReAct 转不顺，别想多 Agent。** 先做一个会用工具、能循环、不失控的单 Agent。

### 必须停下来思考的决策点

**决策点 4：ReAct 循环怎么实现？**
- 亲手理解：Thought（LLM想）→ Action（选工具+参数）→ Observation（执行结果）→ 回到 Thought
- ❗不要直接用框架封装好的黑盒，至少手写一遍核心循环，理解它怎么转的

**决策点 5：三层保护怎么设计？**
- 最大步数（防无限循环）/ 循环检测（防来回调同一工具）/ 异常降级（工具挂了不崩）
- ❗亲手构造失控场景测试每一层。面试官必问"死循环怎么办"

**决策点 6：什么时候用强模型，什么时候用弱模型？**
- 思考：规划/决策用强模型（72B/API），简单执行用弱模型（7B）
- 这是成本意识，记进决策档案

### 必须亲手做的事
1. **亲手默写一遍 ReAct 循环**：不看代码能讲出 think-act-observe 怎么转
2. **亲手看一次完整 trace**：一个任务里 Agent 调了几次工具、每次怎么想的
3. **亲手触发三层保护**：构造死循环、工具失败场景，确认保护生效

### 工程化展现点
- 容错设计（三层保护）
- 状态管理（Agent 的中间状态怎么存）

### 要记录什么
- D-04：ReAct 实现 + 为什么不用 CoT/ToT（结合面试真题：ToT token 三倍成本扛不住）
- D-05：三层保护设计 + 测试case
- D-06：分级用模型的成本策略
- 数据：单 Agent 在测试任务上的成功率、平均工具调用次数

### 自检
- 默写 ReAct 循环
- ReAct / CoT / ToT 区别？你为什么用 ReAct？
- 工具调用失败/死循环怎么办？
- 什么时候用大模型什么时候用小模型？

## 阶段2：记忆系统 + 上下文压缩（约 25h）

### 这个阶段的本质
❗**这是 Agent 项目最值钱的部分。** 没有记忆压缩，Agent 跑几步上下文就爆。这也是面试深挖点。

### 必须停下来思考的决策点

**决策点 7：记忆怎么分层？**
- 工作记忆/短期/长期/共享黑板，各存什么、存哪里、生命周期多长
- ❗要想清楚每一层的边界，不是堆概念

**决策点 8：上下文压缩怎么做到"压缩还不丢信息"？**
- ❗核心设计：四级分层折叠 + 可恢复（原文卸载到文件留路径，摘要是索引不是替代）
- 面试必问"压缩会不会丢信息"，你的答案是"可恢复，按路径取回原文"

**决策点 9：什么时候触发压缩？**
- 思考：按 token 阈值触发，还是按轮数？要有明确策略

### 必须亲手做的事
1. **❗亲手做消融实验**：有记忆压缩 vs 无压缩，跑同样的长任务，对比 token 消耗和任务完成率
   - 这是面试硬通货，"token 节省 XX%，完成率从 XX% 到 XX%"
2. **亲手看上下文增长**：打印每一步的 token 数，亲眼看到不压缩时怎么爆的
3. **亲手验证可恢复**：压缩后，让 Agent 取回某个被折叠的原文，确认能拿回来

### 工程化展现点
- **上下文工程**是 Agent 项目的核心工程能力，这个阶段集中展示
- 消融实验体现你的"数据驱动"和"严谨"

### 要记录什么
- D-07：记忆分层设计
- D-08：四级压缩方案 + 可恢复设计
- D-09：压缩触发策略
- ❗消融实验数据表（有/无各组件的对比）

### 自检
- 记忆为什么要分层？每层存什么？
- 为什么不全量保留上下文？
- 压缩会丢信息吗？怎么保证可恢复？
- 有数据证明记忆压缩有用吗？（消融实验）

## 阶段3：Orchestrator + 多 Agent 编排（约 30h）

### 这个阶段的本质
单 Agent 稳了、记忆有了，现在加多 Agent。**注意：是因为单 Agent 解决不了复杂任务才加多 Agent，不是为了炫技。**

### 必须停下来思考的决策点

**决策点 10：任务怎么规划成 DAG？**
- 思考：为什么用 DAG 不用线性步骤？因为有些子任务能并行（无依赖）
- 亲手看 LLM 规划出的 DAG 合不合理

**决策点 11：多 Agent 怎么通信？**
- ❗关键认知：用共享黑板（数据库）+ Orchestrator 调度，不是 Agent 之间直接喊话
- 面试会问"多 Agent 怎么协作的"

**决策点 12：规划错了怎么办？**
- Plan-and-Execute 的弱点。要有重规划机制：执行中发现计划不合理能动态调整

### 必须亲手做的事
1. **亲手画 Agent 协作图**：哪个 Agent 产出什么、传给谁、怎么汇总
2. **亲手测并行**：构造两个无依赖子任务，确认它们真的并行了（看时间）
3. **亲手做消融**：单 Agent vs 多 Agent 在复杂任务上的对比

### 工程化展现点
- DAG 调度、并行执行、依赖管理（并发能力）
- 系统设计能力（多个组件怎么协同）

### 要记录什么
- D-10：DAG 规划设计 + 为什么不用线性
- D-11：多 Agent 通信机制（共享黑板）
- D-12：重规划机制
- 消融数据：单 vs 多 Agent，成功率和耗时对比，并行加速比

### 自检
- 为什么用多 Agent 不用单 Agent？有数据吗？
- 多 Agent 怎么通信协作？
- 为什么用 DAG？怎么实现并行？
- 规划错了怎么办？

## 阶段4：Critic 反思闭环（约 20h）

### 这个阶段的本质
加质量门控。Critic Agent 审查产出，不合格打回重做。这是 Reflection 模式的应用。

### 必须停下来思考的决策点

**决策点 13：Critic 审查什么维度？**
- 证据充分性、逻辑一致性、是否遗漏反面观点、来源可靠性
- 要有明确的审查标准，不是笼统"看看好不好"

**决策点 14：打回重做怎么避免无限重做？**
- 和 Agent 失控一样，要有重做次数上限

### 必须亲手做的事
1. **亲手构造一个有漏洞的产出**，看 Critic 能不能发现并打回
2. **亲手对比**：有 Critic vs 无 Critic，产出质量差异

### 要记录什么
- D-13：Critic 审查维度设计
- D-14：重做次数控制
- 数据：Critic 带来的质量提升、重做率

### 自检
- Critic 审查哪些维度？
- 这是什么 Agent 模式？（Reflection）
- 怎么防止无限重做？

## 阶段5：流式思考过程 + 前端 + 可观测（约 25h）

### 这个阶段的本质
Agent 项目的演示效果极其重要——**实时展示 Agent 的思考过程**（think-act-observe 流式推送），这是最抓眼球的卖点。

### 必须停下来思考的决策点

**决策点 15：怎么实时展示 Agent 思考？**
- SSE 把每个 Agent 的 thought/action/observation 流式推到前端
- 这是 Deep Research 类产品的标志性体验

**决策点 16：可观测性怎么追踪多 Agent？**
- ❗用 Langfuse 追踪多 Agent 决策树。出错时要能定位是哪个 Agent、哪一步出的问题

### 必须亲手做的事
1. **亲手实现思考过程流式**：让前端实时显示 Agent 在想什么、调什么工具
2. **亲手看一次多 Agent 决策树**：在 Langfuse 里看完整的协作链路
3. **亲手部署 + 录演示**

### 工程化展现点
- 可观测性（多 Agent 决策树回放）——面试讲"哪个 Agent 出错我能定位"
- 流式稳定性

### 要记录什么
- D-15：实时思考过程的实现
- D-16：多 Agent 可观测方案 + 一次真实的问题定位案例

### 自检
- 怎么实时展示 Agent 思考？
- 多 Agent 出错怎么定位是哪个 Agent？

## 总结：两个项目做完，你应该达到的状态

| 能力 | 项目一给你的 | 项目二给你的 |
|------|------------|------------|
| RAG 深度 | 检索/路由/融合/幻觉治理 + 全套数据 | （复用项目一作为工具） |
| Agent 深度 | Agentic 闭环入门 | ReAct/规划/记忆/多Agent/反思 全套 |
| 工程化 | 评估驱动/抽象/可观测/部署 | 工具系统/上下文工程/容错/并发 |
| 讲清楚 | 12+ 条 RAG 决策档案 | 16+ 条 Agent 决策档案 |

**做完两个项目 + 写完决策档案 + 模拟面试练熟，你就从"调包侠"变成了"能讲清楚每个决策的工程师"。这就是 25K+ 的门槛。**

## 最后的话（针对你的情况）

1. **别怕慢**：项目二比项目一难，Agent 概念要花时间消化。慢就是快，搞懂一个概念比抄三段代码值
2. **每阶段都要能讲**：完成一个阶段，对着自检清单自问自答，卡壳就回去补
3. **消融实验是你的核武器**：双非+1年的背景，你拼不过学历和年限，但你能拿出"我做了严谨的消融实验，每个组件贡献多少有数据"——这是很多科班生都做不到的
4. **边做边投**：项目一做到阶段4就可以开始投简历了，用面试反馈调整项目二的重点
