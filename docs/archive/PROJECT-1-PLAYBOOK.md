# 项目一执行手册（给你自己看的）

> 配套文档：`PROJECT-1-AGENTIC-RAG.md`（那份是"做什么"，这份是"你该怎么想、什么时候做什么"）
> 核心目标：让你从"指挥 AI 写代码"升级为"懂工程决策的人"。面试官问的不是代码，是你脑子里的决策过程。

---

## 写在前面：你最大的风险是什么

你现在的模式是 AI coding 帮你把架构实现出来。**这条路最大的陷阱是：代码跑通了，但你不知道为什么这么做。** 面试时一问就垮。

所以这份手册的每一步，都会告诉你：
- **停下来思考的点**（不要让 AI 直接跳过）
- **必须亲手做/亲眼看的事**（不能全交给 AI）
- **要记录什么**（决策档案，面试时的弹药）
- **常见的坑**（你这个阶段最容易踩的）

**铁律：每完成一个模块，你要能脱稿讲出"我为什么这么做、试过什么别的、数据是多少"。讲不出来，就说明这个模块你没真正掌握，回去补。**

## 工程化能力在什么时候展现？（先建立全局认知）

面试官判断"工程化能力"看的是这些信号，它们分布在项目的不同阶段：

| 工程化信号 | 在哪个阶段体现 | 面试官想看到什么 |
|-----------|--------------|----------------|
| 数据处理能力 | 阶段0 | 你能把脏数据变成可用语料，有完整流水线 |
| 抽象/解耦设计 | 阶段1 | 你做了 VectorStore 抽象层，切换后端不用改业务 |
| 评估驱动开发 | 贯穿全程 | 你每次改动都跑评估，有数据支撑，不是凭感觉 |
| 选型决策 | 每个组件引入时 | 你知道为什么选 A 不选 B，有 trade-off 分析 |
| 可观测性 | 阶段4 | 出了问题你能定位，有 trace 有指标 |
| 性能优化 | 阶段4 | 你能说出瓶颈在哪、怎么优化的、提升多少 |
| 异常处理/降级 | 阶段2-3 | 系统坏了不崩，有兜底 |
| 部署运维 | 阶段5 | 容器化、能上线、可监控 |

**关键认知：工程化不是某个阶段的事，是贯穿始终的习惯。** 最重要的两个习惯是：① 每个决策都记录理由；② 每次改动都用数据验证。

## 阶段0：数据流水线 + 评估基线（约 25h）

### 这个阶段的本质
**先有标尺，再谈优化。** 没有评估集，你后面所有"优化"都是自我感动。这也是你练手项目最大的问题——没有基线，说不出提升了多少。

### 你必须停下来思考的决策点

**决策点 1：数据规模定多少？**
- 思考：太少（几百条）暴露不出生产问题，太多（几万条）token 和时间扛不住
- 建议：3K-5K 文档。理由要能讲：「这个规模足够暴露分块、路由、召回的真实问题，又不会让索引构建和评估成本失控」
- ⚠️ 坑：不要一上来追求大数据量，先用 500 条把流程跑通，再扩到 3K

**决策点 2：四个分库怎么分？分块策略怎么定？**
- 思考：法条按"条"分（结构清晰），案例按语义段落分（长文本），这是有理由的差异化
- 这里就是你第一个选型决策，要记进决策档案
- ⚠️ 坑：不要四个库用同一种分块策略，那暴露你没思考文档结构差异

**决策点 3：分块大小（chunk size）定多少？**
- ❗这是必须亲手做实验的地方，不能让 AI 拍个默认值
- 做法：拿同一批文档，分别用 256/512/1024 三种大小建索引，用评估集测 Top-5 命中率
- 记录：哪个大小最好，为什么。这就是面试时"分块大小怎么定的"的标准答案

### 这个阶段必须亲手做的事
1. **亲眼看数据**：打开 10-20 条原始法律文档，理解它的结构。不看数据就做 RAG = 闭眼开车
2. **亲手标注评估集**：80-150 条 query + 期望文档。这个不能全靠 AI 生成，至少亲自审核每一条。这是你整个项目的标尺
3. **亲手跑第一次基线**：得到第一批数字（Top-1/Top-5），哪怕很难看。难看才有优化空间

### 要记录什么（决策档案）
- D-01：为什么选法律域、数据规模定多少、为什么
- D-02：四分库的分块策略差异 + 理由
- D-03：chunk size 实验数据 + 选择
- 第一份基线报告：`docs/REAL-DATA-BASELINE.md`（各项指标的起点）

### 自检：能回答这些问题吗？
- 你的数据从哪来？怎么清洗的？
- 评估集怎么设计的？覆盖了哪些 case 类型？
- chunk size 为什么是这个值？试过别的吗？

## 阶段1：混合检索 + RRF + Rerank + 路由（约 25h）

### 这个阶段的本质
这是 RAG 的核心战场，也是面试问得最深的地方。每引入一个组件，都要用 AB 数据证明它有用。

### 你必须停下来思考的决策点

**决策点 4：先做单路还是直接混合？**
- ❗正确做法：先单独测纯向量、纯 BM25 的基线，再做混合，这样你才有对比数据
- 如果直接上混合，你就没法回答"混合比单路好多少"
- 这是评估驱动开发的典型体现：先建对照组

**决策点 5：融合用 RRF 还是别的？**
- 这是你项目的招牌选型。必须能讲清 max() / 线性加权 / RRF 三者的区别和你的选择理由
- ❗做实验：max() vs RRF 的 Top-1 命中率对比，这个数字是面试镇店之宝
- 记录 RRF 的 k 值怎么定的（网格搜索 k=20/40/60/80）

**决策点 6：Rerank 加不加？**
- 思考：Rerank 提升精度但增加延迟。要测出"提升多少 vs 多慢"，自己判断值不值
- ❗坑：很多人加了 Rerank 但说不出提升数据，等于白加

**决策点 7：什么时候做 Embedding 选型实验？**
- 时机：检索基线稳定后，再做 Embedding 对比（bge-large / bge-m3/ Qwen）
- ⚠️这个需要 GPU，你可以租服务器集中跑一次
- 不是必做项，但做了是加分项（证明你不迷信排行榜，用自己数据说话）

### 这个阶段必须亲手做的事
1. **亲手看坏例**：检索没命中的 query，亲自看为什么没命中，归类（5类坏例）。这个过程比看指标更长功力
2. **亲手设计 AB 实验**：保留旧方法（max），新增 RRF，能切换对比。这是工程化的体现
3. **亲手画对比表**：把所有配置的指标列成表（纯向量/纯BM25/max/RRF/RRF+Rerank）

### 工程化展现点
- **抽象层设计**：`VectorStore` 接口，Chroma/Qdrant 可切换。面试讲"我做了门面层，切换后端零成本"
- **配置化**：融合方式、k 值、top_k 都做成配置，不写死。体现可维护性

### 要记录什么
- D-04：融合策略选型（max vs 线性 vs RRF）+ 数据
- D-05：RRF 的 k 值调优过程
- D-06：Rerank 的收益/成本权衡
- D-07（可选）：Embedding 对比实验 → `docs/EMBEDDING-DECISION.md`
- 坏例分析报告（5类分布）

### 自检
- 混合检索为什么比单路好？什么时候单路反而好？
- RRF 的原理？k 参数的作用？为什么不用线性加权？
- Rerank 和检索的区别？它能召回检索没捞到的吗？（不能）
- 你的 Embedding 怎么选的？排行榜可信吗？

## 阶段2：Agentic 闭环改造（约 30h）

### 这个阶段的本质
这是从"练手项目"质变为"能打的项目"的关键。把直线流水线改成 LLM 决策的闭环。**这是面试最大的加分项，也是你目前最大的短板。**

### 你必须停下来思考的决策点

**决策点 8：哪些环节让 LLM 决策？**
- 思考：Router（要不要检索/检索哪个库）、Grader（够不够）是决策点
- ❗核心认知：Agent 和 Pipeline 的区别就是"谁决定下一步"。你要能清楚说出你的系统里 LLM 在哪些点做了决策
- 这是面试致命题，必须想透

**决策点 9：终止条件怎么设计？**
- ❗关键认知（来自2026面试真题）：终止由"上下文充分性"决定，不是固定检索次数
- 你要能讲：简单问题1次够，复杂问题可能3次，由 Grader 判断
- 同时要有硬上限（max_iter=3）防止失控

**决策点 10：Grader 阈值怎么定？**
- ❗亲手做：在评估集上画 precision-recall 曲线，选 F1 最高点
- 不能拍脑袋定 0.7，要有数据依据

### 这个阶段必须亲手做的事
1. **亲手画状态机**：在纸上/白板画出 LangGraph 的节点和流转，理解每个状态转移的条件。不理解状态机，你就只是在抄代码
2. **亲手测自纠正**：构造一个"第一次检索失败、改写后成功"的 case，亲眼看到闭环工作
3. **亲手触发三层保护**：故意构造死循环场景，验证保护机制生效

### 工程化展现点
- **状态管理**：LangGraph 的 state 设计（messages/query/docs/grade/iterations）
- **三层保护**：最大迭代/循环检测/异常降级。面试讲"我考虑了 Agent 失控的三种情况"
- **可解释性**：每次决策记录到 trace，能回放为什么这么走

### 要记录什么
- D-08：哪些环节 LLM 决策 + 为什么（Agent vs Pipeline 的体现）
- D-09：终止条件设计（上下文充分性 vs 固定次数）
- D-10：Grader 阈值标定过程
- D-11：三层保护设计
- 对比数据：Agentic 改造前后，复杂/多跳问题的准确率对比

### 自检
- RAG+Chat 算 Agent 吗？为什么？（不算，固定流水线）
- 你的系统里 LLM 在哪些点做决策？
- 终止条件是什么？为什么不用固定次数？
- 自纠正会死循环吗？怎么防？

## 阶段3：幻觉检测 + 引用溯源升级（约 20h）

### 这个阶段的本质
法律场景幻觉零容忍，这是你项目的差异化卖点。把练手项目的"字符串重叠"升级为语义级。

### 你必须停下来思考的决策点

**决策点 11：幻觉检测用什么方法？**
- 选项：字符串匹配（你现在的，太弱）/ NLI 模型 / LLM 自检
- 思考权衡：精度 vs 成本。讲清楚你为什么选这个
- ❗这是从"0层深度"升级的关键，面试官会专门问引用溯源

**决策点 12：检测到幻觉怎么处理？**
- 选项：带反馈重新生成 / 标记"请核实" / 直接拒答
- 要有分级策略，不是一刀切

### 必须亲手做的事
1. **亲手标注幻觉**：在评估集上人工判断每条回答有没有幻觉，算出基线幻觉率
2. **亲手对比**：升级检测方法前后，幻觉率从多少降到多少

### 要记录什么
- D-12：幻觉检测方法选型 + 理由
- D-13：引用溯源从字符串升级到语义级的过程
- 幻觉率优化数据（从 XX% 到 XX%）

### 自检
- 你怎么衡量幻觉率？
- 引用溯源怎么做的？比字符串匹配强在哪？
- 检测到幻觉怎么处理？

## 阶段4：缓存 + 可观测性（约 15h）

### 这个阶段的本质
这是"生产级"和"demo"的分水岭。能不能上线、出问题能不能查，看这里。

### 你必须停下来思考的决策点

**决策点 13：三级缓存的语义缓存阈值怎么定？**
- ❗亲手做：相似度阈值太高没命中率，太低返回不相关结果。在测试集上找平衡点
- 要能讲"缓存一致性"：知识库更新时怎么失效缓存

**决策点 14：可观测性看什么指标？**
- 系统指标（Prometheus）：QPS、延迟分布、错误率
- LLM 指标（Langfuse）：token 消耗、各节点延迟、决策树回放

### 必须亲手做的事
1. **亲手看一次完整 trace**：从请求进来到返回，每一步的输入输出和耗时。这是你"排查问题"能力的来源
2. **亲手找瓶颈**：看延迟分布，哪一步最慢，怎么优化

### 工程化展现点
- 这整个阶段就是工程化能力的集中展示
- 面试讲"线上出问题我怎么排查"→ 拿 trace 说话

### 要记录什么
- D-14：缓存策略设计 + 阈值标定 + 一致性方案
- D-15：可观测性方案 + 一次真实的瓶颈定位案例
- 性能数据：P99 延迟、缓存命中率、平均 token 成本

### 自检
- 缓存阈值怎么定？缓存一致性怎么保证？
- 系统出问题你怎么排查？
- 性能瓶颈在哪？怎么优化的？

## 阶段5：流式 + 前端 + 部署（约 20h）

### 这个阶段的本质
让项目"能演示"。面试时能掏出一个跑起来的系统，比说一万句都强。

### 你必须停下来思考的决策点

**决策点 15：流式用 SSE 还是 WebSocket？**
- 思考：问答是服务端→客户端单向，SSE 足够且更简单。要能讲为什么不用 WebSocket
**决策点 16：部署用 Docker Compose 还是 K8s？**
- 个人项目 Docker Compose 够了，但要能讲"如果要上生产/扩容，怎么迁到 K8s"

### 必须亲手做的事
- **亲手部署一次**：从零在租的服务器上用 Docker 跑起来。部署踩的坑就是面试的谈资
- **亲手录个演示视频**：放简历里/面试展示

### 工程化展现点
- 容器化、环境隔离、一键部署
- 流式输出的稳定性（取消/超时/断线重连）——这是面试复盘里被点名的加分项

### 自检
- SSE vs WebSocket 怎么选？
- 流式输出怎么处理取消/超时/断线？
- 怎么部署的？要扩容怎么办？

## 阶段6：评估总结 + 决策档案 + 面试稿（约 15h）

### 这个阶段的本质
把前面所有的数据、决策、踩坑整理成面试弹药。**项目做得好不等于面得好，这一步把项目变成 offer。**

### 必须亲手做的事
1. **整理完整指标表**：所有模块优化前后的对比数据汇总成一张表
2. **写决策档案**：12+ 条决策，每条按"背景/备选/为何选/代价/数据/追问"结构
3. **录音模拟面试**：对着决策档案讲，录下来回放，听自己哪里卡壳
4. **整理 STAR-L 简历描述**：每条 bullet 带数字

### 自检（终极）
对着 `RAG-AGENT-KNOWLEDGE-MAP.md` 末尾的自检清单，逐条过。打不了勾的就是盲区。

## 总结：你在每个阶段的角色转变

| 阶段 | AI 干什么 | 你干什么（关键） |
|------|----------|----------------|
| 0 | 写数据流水线代码 | 看数据、标评估集、做 chunk 实验、定规模 |
| 1 | 实现检索/融合/路由 | 设计 AB 实验、看坏例、做选型决策、画对比表 |
| 2 | 写 LangGraph 状态机 | 理解状态机、设计决策点和终止条件、标定阈值 |
| 3 | 实现幻觉检测 | 标注幻觉、选检测方法、对比数据 |
| 4 | 接缓存和监控 | 看 trace、找瓶颈、定缓存阈值 |
| 5 | 写前端和部署脚本 | 亲手部署、录演示、想扩容方案 |
| 6 | 帮你整理文档 | 写决策档案、模拟面试、查漏补缺 |

**一句话：AI 负责"实现"，你负责"决策、验证、讲清楚"。这三件事就是 25K+ 工程师和调包侠的区别。**
