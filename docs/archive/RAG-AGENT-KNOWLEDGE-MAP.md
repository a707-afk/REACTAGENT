# RAG / Agent 工程全域知识地图

> 这不是项目优化指南，这是你作为 RAG/Agent 工程师的**完整认知体系**。
> 面试官的问题不会只围绕你的项目问，他会问你对整个领域的理解。
> 目标：让你对任何角度的追问，都有思考框架可以接住。

---

## 全局认知：一个 RAG 系统的六层架构

```
用户问题
   ↓
┌──────────────────────────────────────────────┐
│ 第一层：理解层（Query Understanding）          │
│   问题分类 / 意图识别 / Query Rewrite / 扩展  │
└──────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────┐
│ 第二层：检索层（Retrieval）                    │
│   稀疏 / 稠密 / 混合 / 路由 / 过滤           │
└──────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────┐
│ 第三层：后处理层（Post-Retrieval）             │
│   Rerank / 压缩 / 去重 / 冲突检测 / 门控     │
└──────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────┐
│ 第四层：生成层（Generation）                   │
│   Prompt 工程 / 引用注入 / 流式 / 格式控制    │
└──────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────┐
│ 第五层：验证层（Verification）                 │
│   幻觉检测 / 引用溯源 / 事实核查 / 安全护栏   │
└──────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────┐
│ 第六层：反馈层（Feedback & Learning）          │
│   用户反馈 / 日志分析 / 持续评估 / 数据飞轮   │
└──────────────────────────────────────────────┘

横切关注点（贯穿所有层）：
  - 可观测性（tracing / metrics / logging）
  - 缓存（多级）
  - 成本控制（token / 延迟 / 计算资源）
  - 评估体系（离线 + 在线）
  - 安全（prompt injection / data leakage / PII）
```

## 第一层：理解层（Query Understanding）

### 1.1 为什么需要理解层？
用户输入的问题往往不是最好的检索条件。口语化、模糊、多意图、省略上下文。
如果直接拿原始问题去检索，召回质量天花板就在那里，后面怎么优化都补不回来。

**面试认知**：理解层是整个 RAG 的天花板。检索做得再好，问题理解错了，召回的东西就是错的。

### 1.2 问题分类 / 意图识别
| 技术方案 | 原理 | 优势 | 劣势 | 适用场景 |
|----------|------|------|------|---------|
| 关键词规则 | 正则 / 关键词命中 | 快、可解释、零成本 | 覆盖率低、维护成本随规则数爆炸 | 已知意图类型少（<10类）且有明确关键词 |
| 传统 ML 分类器 | TF-IDF + SVM/LR | 快、资源少 | 需要标注数据、泛化差 | 有足够标注数据时 |
| Embedding 相似度 | 把意图做成 anchor embedding，算距离 | 少量样本可用、零训练 | 边界模糊时准确率不够 | 意图类型 10-50 个 |
| LLM 分类 | 直接让 LLM 做 few-shot 分类 | 泛化强、无需训练 | 延迟高、成本高 | 兜底方案 / 高价值场景 |
| 级联方案 | 规则 → Embedding → LLM | 综合平衡 | 实现复杂 | **生产推荐** |

**面试关键认知**：
- 不存在最好的方案，只有最适合你场景的方案
- 级联的核心是"快路径处理简单case，慢路径兜底难case"
- 面试官会问"为什么不直接全用 LLM？"→ 因为成本和延迟，90% 的简单问题不值得调 LLM

### 1.3 Query Rewrite / 查询改写
| 技术方案 | 原理 | 什么时候用 | 什么时候不用 |
|----------|------|-----------|-------------|
| 同义词替换 | 维护同义词表 | 领域术语固定 | 开放域 |
| 规则模板 | 正则提取关键信息重组 | 格式化查询（如"XX条"） | 自然语言查询 |
| HyDE | 先让 LLM 生成假设文档，用假设文档做检索 | 稠密检索效果差时 | 增加一次 LLM 调用的延迟/成本 |
| Query Expansion | 生成多个变体查询，分别检索后合并 | 召回率不够时 | 精度已经够高不想引入噪声 |
| Step-back Prompting | 先让 LLM 提出更高层次的问题 | 过于具体的问题检索不到时 | 问题本身已经是高层次的 |
| Multi-Query | LLM 生成多角度查询 | 问题涉及多个方面 | 简单直接的问题 |

**面试关键认知**：
- Query Rewrite 的价值必须用 AB 数据证明：开启/关闭后的召回率差值
- HyDE 不是万能的 — 当 LLM 对领域不了解时，生成的假设文档本身就是错的，会把检索带偏
- 面试官常问"Query Rewrite 有没有副作用？"→ 有，可能改写后偏离原意，需要兜底机制（如原始查询+改写查询都检索，取并集）

### 1.4 多轮对话中的理解层
单轮好做，多轮才是难点：
- **指代消解**："这个案子"指的是上文第几轮提到的哪个案子？
- **上下文补全**：第三轮问"那赔偿标准呢？" → 需要补全为"XX事故类型下的赔偿标准"
- **意图转移**：用户从问法条变成问案例，需要切换检索策略

**面试关键认知**：多轮场景下 Query Rewrite 的难度比单轮高一个数量级，这也是为什么需要 Agent 记忆

## 第二层：检索层（Retrieval）

### 2.1 检索范式全景
| 范式 | 代表技术 | 原理 | 优势 | 劣势 |
|------|---------|------|------|------|
| 稀疏检索 | BM25 / TF-IDF | 词频统计 + 倒排索引 | 精确匹配强、可解释、零GPU | 语义不匹配（"欠钱"vs"债务"）|
| 稠密检索 | 向量 Embedding | 语义空间距离 | 语义匹配、跨语言 | 精确匹配差（法条编号）、需要GPU |
| 混合检索 | BM25 + Vector | 两路召回合并 | 互补 | 需要融合策略 |
| 学习排序 | Cross-Encoder Rerank | 把 query-doc 对一起编码打分 | 精度高 | 只能处理候选集、不能检索全库 |
| 知识图谱检索 | SPARQL / 图遍历 | 结构化关系推理 | 多跳推理 | 构建成本极高 |
| 结构化检索 | SQL / Elasticsearch | 精确字段查询 | 结构化数据精确查 | 非结构化文本不适用 |

**面试关键认知**：
- 没有银弹，生产系统几乎都是混合检索
- 面试官会问"稠密检索什么时候不如稀疏？"→ 精确关键词匹配场景（编号、人名、法条号）
- Rerank 不是检索方案，是后处理方案 — 它不能召回检索阶段没捞到的文档

### 2.2 Embedding 模型选型
| 维度 | 需要考虑的因素 |
|------|--------------|
| 性能 | 在**你的数据**上的 Top-K 命中率，不是排行榜分数 |
| 维度 | 影响存储成本和检索速度。768 vs 1024 vs 2560 |
| 推理速度 | CPU 能跑还是必须 GPU？单条延迟多少？ |
| 多语言 | 只需要中文？还是中英混合？ |
| 最大长度 | 512 token 够不够？法律文本可能很长 |
| 微调可能 | 如果通用模型不够，能不能在领域数据上微调？ |
| 开源/闭源 | 闭源有 API 调用成本，开源有部署成本 |

**面试必须会回答的问题**：
- "你怎么选的 Embedding 模型？" → 不要说"排行榜上好"，要说"在我的 XX 条测试集上跑了对比，A 模型 Top-5 XX%，B 模型 XX%"
- "为什么不微调？" → 微调需要大量配对数据（query-passage pairs），我的标注数据只有 XX 条，不够支撑微调。如果有 1 万条以上标注数据，会考虑用对比学习微调
- "Embedding 维度高了会怎样？" → 存储翻倍、ANN 检索变慢、有维度灾难风险。但高维度理论上能编码更多语义。实际需要用数据验证高维度的收益是否能覆盖成本

### 2.3 分块策略（Chunking）
| 策略 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| 固定长度 | 每 N 个 token 切一刀 | 简单 | 可能切断语义 | 纯文本无结构 |
| 固定长度 + 重叠 | 每 N token 切，前后重叠 M | 减少信息丢失 | 冗余存储 | 通用 |
| 按标题/章节 | 用 Markdown 标题做分割点 | 保留文档结构 | 块大小不均匀 | 结构化文档 |
| 语义分块 | 用 Embedding 相似度检测语义断点 | 语义完整 | 慢、依赖模型 | 长文档 |
| 递归分割 | 先按大结构分，再按小结构分，逐级细化 | 灵活 | 参数多 | LangChain 默认方案 |
| 父子块 | 小块检索，命中后返回父块（更大上下文） | 检索精度高 + 生成上下文够 | 实现复杂 | **生产推荐** |
| 命题分块 | 先用 LLM 把文档分解为独立命题 | 每块一个清晰论点 | LLM 调用成本高 | 高价值文档 |

**面试关键认知**：
- **分块是 RAG 中被低估但影响最大的环节之一**。块太大，检索精度差（噪声多）；块太小，上下文不够 LLM 理解
- 父子块是目前公认的平衡方案：用小块（256 token）做精准检索，命中后返回父块（1024 token）给 LLM，既准又有上下文
- 面试官会问"你的分块大小怎么定的？"→ 不能说"默认值"，要说"在我的法律文本上实验了 256/512/1024，测了 Top-5 命中率和 LLM 回答质量，发现 512 是最佳平衡点"

### 2.4 混合检索融合
| 融合策略 | 原理 | 优势 | 劣势 |
|----------|------|------|------|
| 取最大值 max() | 同一文档取两路最高分 | 简单 | 偏向单路强势 |
| 线性加权 αV + (1-α)B | 手动定权重 | 可调 | 最佳 α 随查询类型变 |
| RRF 1/(k+rank) | 基于排名融合 | 对分数尺度鲁棒 | 忽略分数差距 |
| 学习权重 | 用标注数据训练融合模型 | 自适应 | 需要大量标注数据 |
| Conditional | 根据查询类型决定只用哪一路或两路 | 最优 | 需要查询分类器 |

**面试关键认知**：
- RRF 是目前业界默认选择，因为它不需要对齐两路的分数尺度
- 但 RRF 也不完美 — 它把排名第1和排名第2的差距，和排名第50和排名第51的差距同等对待
- 高级方案是 Conditional：先判断查询类型，精确查询只走 BM25，语义查询只走向量，模糊的走混合

### 2.5 向量数据库
| 方案 | 定位 | 性能 | 过滤 | 分布式 | 适用 |
|------|------|------|------|--------|------|
| Chroma | 嵌入式 | 万级 OK | 基础 metadata | 不支持 | 原型/小规模 |
| Qdrant | 专用向量库 | 百万级 | 载荷过滤强 | 支持 | 中大规模生产 |
| Milvus | 分布式向量库 | 十亿级 | 支持 | 原生分布式 | 大规模 |
| Weaviate | 向量+对象 | 百万级 | GraphQL | 支持 | 需要对象存储场景 |
| pgvector | PG 扩展 | 百万级 | SQL 全功能 | PG 集群 | 已有 PG 生态 |
| Elasticsearch 8+ | 混合搜索引擎 | 大规模 | DSL 全功能 | 原生分布式 | 已有 ES 生态 |
| FAISS | 库（非数据库） | 超大规模 | 不支持 | 不支持 | 纯 ANN 计算 |

**面试关键认知**：
- 选向量库不是选"最强的"，是选"最适合你团队和场景的"
- 面试官会问"为什么不用 Milvus？"→ 我的数据量是 X 千条，Milvus 是十亿级分布式系统，引入它的运维成本远超收益
- 面试官会问"Chroma 能用于生产吗？"→ 千级文档可以，万级以上建议切 Qdrant。关键是代码要做好抽象，切换成本低
- **ANN 算法也是考点**：HNSW（图搜索，精度高但内存大）vs IVF（聚类后搜索，内存小但精度略低）vs PQ（量化压缩，牺牲精度换内存）

## 第三层：后处理层（Post-Retrieval）

### 3.1 Rerank
| 方案 | 原理 | 精度 | 速度 | 成本 |
|------|------|------|------|------|
| Cross-Encoder | 把 query+doc 拼接输入 BERT 类模型打分 | 最高 | 慢（O(n)次推理）| GPU |
| ColBERT | 延迟交互，token 级别相似度 | 高 | 较快 | GPU |
| LLM Rerank | 让 LLM 对候选文档排序 | 高 | 最慢 | API 成本 |
| Cohere Rerank API | 商业 API | 高 | 快 | 按调用收费 |
| bge-reranker | 开源 Cross-Encoder | 高 | 中 | 本地 GPU |

**面试关键认知**：
- Rerank 的输入是检索阶段已召回的 Top-K，它不能召回检索没捞到的
- 所以如果检索阶段的召回率（Recall@K）就很低，Rerank 再强也没用
- 面试官会问"Rerank 和检索有什么区别？"→ 检索是从百万文档中选 Top-100，Rerank 是从 Top-100 中精排出 Top-5。检索要求速度（ANN），Rerank 要求精度（Cross-Encoder）
- "Rerank 提升了多少？"→ 必须有数据：Rerank 前 Top-5 命中率 XX%，Rerank 后 XX%

### 3.2 上下文压缩
召回了 10 个 chunk，全部塞进 prompt 太长 → 需要压缩
| 方案 | 做法 | 优势 | 劣势 |
|------|------|------|------|
| 截断 | 只取 Top-K | 简单 | 可能丢关键信息 |
| LLM 摘要 | 让 LLM 压缩每个 chunk | 保留关键信息 | 额外 LLM 调用 |
| 相关句提取 | 只保留和 query 相关的句子 | 精准 | 可能断章取义 |
| Long Context | 用支持长上下文的 LLM | 不需要压缩 | Lost-in-the-middle 问题 |

**面试关键认知**：
- "Lost in the middle"：LLM 对 prompt 头部和尾部的信息关注度高，中间的容易忽略。所以把最相关的 chunk 放在最前面和最后面
- 上下文窗口大 ≠ 能有效利用。128K 窗口不意味着你该塞 128K 内容

### 3.3 冲突检测与去重
同一个问题，不同 chunk 可能给出矛盾信息（比如不同年份的法规），需要检测和处理

## 第四层：生成层（Generation）

### 4.1 Prompt 工程
| 技巧 | 适用场景 |
|------|---------|
| System Prompt 定角色 | 所有场景（设定领域专家身份）|
| Few-shot 示例 | 输出格式要求严格时 |
| Chain-of-Thought | 需要推理的复杂问题 |
| 引用格式指令 | 要求 LLM 标注来源（[1][2]）|
| 拒答指令 | 当检索结果不足时要求说"不确定" |
| 结构化输出 | JSON / Markdown 格式控制 |

**面试关键认知**：
- Prompt 工程不是"写几句话"，是系统工程：版本管理、AB 测试、效果评估
- 面试官会问"你的 Prompt 怎么迭代的？"→ 有版本号，每次改动都跑评估集对比，确认没有 regression
- "幻觉怎么控制？"→ Prompt 层面：明确要求"只基于提供的文档回答，不确定就说不确定"；系统层面：引用溯源验证

### 4.2 流式输出（SSE/WebSocket）
- 为什么需要：LLM 生成慢，不流式用户等 5-10 秒没反馈
- SSE vs WebSocket：SSE 单向足够（服务端→客户端），WebSocket 双向但更复杂
- 流式引用：难点是引用标注在生成过程中还不确定，需要后处理

### 4.3 LLM 选型
| 维度 | 考虑因素 |
|------|---------|
| 开源 vs 闭源 | 数据安全（法律数据敏感）、成本、定制化需求 |
| 参数量 | 7B 本地可跑 / 70B 需要 GPU 集群 / API 调用无需关心 |
| 上下文窗口 | 4K / 8K / 32K / 128K — 影响能塞多少 chunk |
| 推理能力 | 简单 QA 不需要强推理，复杂法律分析需要 |
| 中文能力 | 法律场景中文能力是硬要求 |
| 成本 | token 单价 × 日均调用量 = 月成本 |

**面试关键认知**：
- "为什么不用 GPT-4？"→ 要有理由：成本 / 数据出境合规 / 延迟 / 中文能力对比
- "7B 模型够用吗？"→ 看任务复杂度。简单问答足够，复杂推理不够。用评估集量化：7B 准确率 XX%，API 模型 XX%，差距在不在可接受范围内

## 第五层：验证层（Verification）

### 5.1 幻觉检测
| 方案 | 原理 | 精度 | 成本 |
|------|------|------|------|
| 字符串匹配 | 检查回答中的关键信息是否出现在检索文档中 | 低（同义词无法匹配） | 零 |
| NLI 模型 | 自然语言推理判断"回答"是否被"文档"蕴含 | 中高 | 一次推理 |
| LLM 自检 | 让另一个 LLM 判断回答是否有文档支撑 | 高 | 额外 LLM 调用 |
| 引用溯源 | 回答每句话标注来源，验证来源存在且支持该句 | 高 | 实现复杂 |
| 多路验证 | 多次生成比较一致性，不一致的地方可能是幻觉 | 高 | 多次 LLM 调用 |

**面试关键认知**：
- 幻觉率是 RAG 系统最重要的指标之一
- "你怎么衡量幻觉率？"→ 在评估集上人工标注每条回答是否有幻觉，统计比例
- 生产系统中幻觉率从 20% 降到 5% 是实打实的价值

### 5.2 安全护栏
| 风险 | 攻击方式 | 防御 |
|------|---------|------|
| Prompt Injection | 用户输入中嵌入指令覆盖系统 prompt | 输入清洗 + 角色隔离 + 输出检测 |
| Jailbreak | 诱导 LLM 突破安全限制 | 行为检测 + 多层过滤 |
| 数据泄露 | 通过提问套出训练数据或知识库内容 | 输出过滤 + 权限控制 |
| PII 泄露 | 回答中包含个人隐私信息 | PII 检测 + 脱敏 |

**面试关键认知**：
- 安全不是加个过滤器就行，是要分层防御（input → process → output）
- 法律场景尤其敏感，当事人信息、案件细节都可能涉及隐私

## 第六层：反馈层（Feedback & Learning）

### 6.1 评估体系
| 评估类型 | 方式 | 指标 | 频率 |
|----------|------|------|------|
| 离线评估（检索） | 标注测试集 + 自动跑 | Top-1/5/10 命中率, MRR, nDCG | 每次改动 |
| 离线评估（生成） | 人工标注 + LLM 评分 | 准确率, 完整性, 幻觉率, 引用准确率 | 每次改动 |
| 在线评估 | 用户行为分析 | 点赞率, 追问率, 跳出率, NPS | 持续 |
| AB 测试 | 同一查询两个版本对比 | 相对提升 | 重大改动时 |
| 坏例分析 | 分类 Top-K 没命中的原因 | 按类型统计分布 | 每次评估后 |

**面试关键认知**：
- "你怎么知道你的系统好不好？"→ 不能只说"试了几个问题感觉还行"
- 需要有标注测试集 + 自动化评估脚本 + 坏例归因分类
- 坏例通常分这几类：
  1. 文档不存在（知识库没收录）
  2. 分块切碎了（答案跨块）
  3. 语义鸿沟（口语 vs 术语）
  4. 路由错误（问法条但路由到了案例库）
  5. 排序错误（文档召回了但排名太低被截断）

### 6.2 数据飞轮
```
用户提问 → 系统回答 → 用户反馈（满意/不满意）
                ↓
         不满意的记录为坏例
                ↓
         分析坏例原因 → 针对性优化
                ↓
         优化后加入测试集防回退
                ↓
         下一轮评估 → 持续改进
```

**面试关键认知**：数据飞轮是让系统越来越好的机制，不是一次性优化

## Agent 知识体系

### A.1 Agent 的本质定义
Agent ≠ Chatbot ≠ Pipeline

| 概念 | 定义 | 关键特征 |
|------|------|---------|
| Chatbot | 一问一答 | 无状态、无工具 |
| Pipeline | 固定流程 A→B→C→D | 有工具但路径固定 |
| **Agent** | LLM 自主决策调用什么工具、何时结束 | **自主决策、动态路径** |

**面试致命问题**：如果你的系统路径是固定的（policy→retrieve→gate→draft→finalize），面试官说"这不是 Agent，这是 Pipeline"，你答不上来就挂了。

### A.2 Agent 架构范式
| 范式 | 流程 | 优势 | 劣势 | 适用 |
|------|------|------|------|------|
| ReAct | Think→Act→Observe→Think→... 循环 | 灵活、可解释 | 可能死循环 | **通用推荐** |
| Plan-and-Execute | 先生成完整计划，再逐步执行 | 全局最优 | 计划可能错、修改代价高 | 复杂多步任务 |
| Reflexion | 执行后反思、修改策略再试 | 自我纠错 | 多次调用成本高 | 高要求任务 |
| Multi-Agent | 多个 Agent 协作 | 任务分解 | 协调复杂 | 大型系统 |
| Function Calling | LLM 原生工具调用接口 | 标准化 | 依赖 LLM 能力 | OpenAI 生态 |

### A.3 工具设计
工具对 Agent 来说就像 API 对开发者：

```json
{
  "name": "search_statute",
  "description": "检索法律条文。当用户问到具体法条、法律规定时使用",
  "parameters": {
    "query": {"type": "string", "description": "检索关键词"},
    "top_k": {"type": "integer", "default": 5}
  }
}
```

**面试关键认知**：
- 工具的 description 质量直接决定 LLM 选择工具的准确率
- 工具要有清晰的边界：search_statute vs search_case vs search_interpretation，不要一个万能工具
- 面试官会问"工具调用失败了怎么办？"→ 三层保护：最大迭代、循环检测、异常回退

### A.4 记忆系统
| 记忆类型 | 生命周期 | 存储 | 用途 |
|----------|---------|------|------|
| 工作记忆 | 当前会话 | Prompt 内 | 当前对话上下文 |
| 短期记忆 | 跨几轮 | 缓存/摘要 | 最近的对话历史 |
| 长期记忆 | 跨会话 | 数据库/文件 | 用户偏好、历史总结 |
| 程序性记忆 | 永久 | 代码/配置 | 系统能力（工具、规则） |

**上下文压缩策略**：
| 策略 | token 消耗 | 信息保留 | 适用 |
|------|-----------|---------|------|
| 全量保留 | 线性增长 | 100% | 短对话 |
| 滑动窗口 | 固定 N 轮 | 丢远期 | 简单场景 |
| 摘要压缩 | 恒定 | 依赖摘要质量 | 中等对话 |
| 分层折叠 | 按需 | 可恢复 | **长对话推荐** |

**面试关键认知**：
- "为什么不全部保留？"→ token 成本线性增长，128K 窗口虽然大但真塞满了 LLM 的注意力会分散
- "摘要会不会丢信息？"→ 会，所以保留原始记录的引用路径，摘要是索引不是替代
- 做消融实验：有记忆 vs 无记忆的任务完成率对比，这是面试硬数据

## 横切关注点

### H.1 可观测性（Observability）
| 层面 | 工具 | 作用 |
|------|------|------|
| Logging | 结构化日志（JSON） | 问题排查 |
| Metrics | Prometheus + Grafana | 性能监控（延迟/吞吐/错误率）|
| Tracing | OpenTelemetry / Langfuse / LangSmith | 请求全链路追踪 |
| LLM 专用 | Langfuse / Phoenix | token 消耗、LLM 延迟、prompt 版本管理 |

**面试关键认知**：
- "你怎么排查线上问题？"→ 每个请求有 trace_id，能看到完整链路：问题理解→检索→重排→生成→验证，每步的输入输出和延迟
- 没有可观测性的系统，出了问题就是盲人摸象

### H.2 缓存
| 级别 | 策略 | 命中条件 | 节省 |
|------|------|---------|------|
| L1 精确匹配 | 完全相同的查询 | Hash 一致 | ~100% 延迟 |
| L2 语义缓存 | Embedding 相似度 > 阈值 | 近似查询 | ~70% 延迟 |
| L3 LLM 判断 | 让 LLM 判断是否可复用 | 语义等价 | 需要 LLM 调用但省检索 |

**面试关键认知**：
- 缓存不只是加速，还能省钱（减少 LLM 调用）
- 面试官会问"语义缓存的阈值怎么定？"→ 太高（0.99）没命中率，太低（0.8）容易返回不相关缓存。在测试集上画 precision-recall 曲线，选 F1 最高的阈值
- "缓存一致性怎么保证？"→ 知识库更新时需要清除相关缓存，否则返回过期信息

### H.3 成本控制
| 成本来源 | 优化方式 |
|----------|---------|
| LLM token | 缓存、上下文压缩、小模型处理简单问题 |
| Embedding 计算 | 批量处理、增量索引（新文档才计算）|
| 向量存储 | 量化压缩（PQ）、分层存储 |
| GPU | 按需使用、选对模型尺寸 |

### H.4 部署与运维
| 话题 | 关键点 |
|------|--------|
| 容器化 | Docker 是基础，K8s 是生产级 |
| CI/CD | 代码提交 → 自动测试 → 评估集回归 → 部署 |
| 灰度发布 | 新版本先导流 10% 用户，观察指标 |
| 限流 | Token Bucket / 滑动窗口，防恶意调用和成本失控 |
| 高可用 | 多副本 + 负载均衡 + 自动扩缩 |

## 面试元能力：怎么回答你不知道的问题

### 框架 1：承认 + 推理 + 延伸
> "这个我没有直接实践过，但根据我对 XX 的理解，我认为 YY，因为 ZZ。如果有机会我会通过 AA 方式验证。"

### 框架 2：类比已知
> "这个场景我没遇到过，但和 XX 有相似之处。在 XX 场景下我们是这样解决的，我猜这个场景可以类似地..."

### 框架 3：问反问
> "这是个好问题。在您的场景中，数据规模大概是什么量级？因为不同量级的解决方案差异很大。"

### 绝对禁区
- 编造经验（"我们生产环境 QPS 一万"→ 你一追问就穿帮）
- 背八股不理解（"RAG 就是检索增强生成"→ 面试官问下一个问题你接不住）
- 只说技术不说 trade-off（"我们用了 RRF"→ "为什么不用线性加权？"→ 卡壳）
