# 项目深度打磨战略指南

> 核心原则：**面试不考你知不知道，考你想没想清楚。**
> 每个技术决策 = 背景 + 备选方案 + 为什么选它 + 放弃了什么 + 量化结果 + 追问预案

---

## 一、你的项目现状诊断

### 能扛住追问的（2层+）
- 混合检索 BM25+Vector（但融合用 max() 不是 RRF，没有 AB 数据）
- Query Rewrite 启发式+LLM（但缺开/关前后的召回率差值）
- 域路由 3阶段：规则+embedding+LLM（架构好但没法律域验证数据）
- Rerank（有但缺量化提升数据）
- 行为护栏/策略引擎 3层（有但缺拦截率/误拦率）

### 面试必穿帮的（0-1层）
- **Agent 是直线流** → policy→retrieve→gate→draft→finalize，没有 ReAct/工具路由
- **引用溯源** → citation_verify.py 只是字符串滑窗重叠匹配，不是语义级
- **记忆/上下文压缩** → 完全没有
- **缓存** → 完全没有
- **评估体系** → 只有检索 Top-1/Top-5，护栏/路由/Agent/端到端全没量化

## 三、必须准备的 5 个选型对比（面试核心）

### 对比 1：混合检索融合策略
| 方案 | 原理 | 优势 | 劣势 | 适用场景 |
|------|------|------|------|---------|
| max() | 取同一文档在两路中的最高分 | 实现简单 | 偏向单路强势结果，另一路的信号被忽略 | 快速原型 |
| 固定权重 α·V + (1-α)·B | 线性加权 | 可调 | 不同问题的最佳 α 差异大，一个权重不适配所有场景 | 数据分布稳定时 |
| **RRF** 1/(k+rank) | 基于排名融合，不依赖分数绝对值 | 对不同分数尺度鲁棒，不需要调权重 | 忽略了分数差距信息 | **生产推荐** |

**你要能讲**：「我一开始用 max()，发现法条号精确查询时 BM25 排第一但向量路排第十几，max() 取了向量分导致法条号查询 Top-1 命中率低。改成 RRF 后 Top-1 从 XX% 提升到 XX%，因为 RRF 基于排名融合，BM25 排第一和向量排第一同等重要。」

### 对比 2：Embedding 模型
| 模型 | 维度 | 参数量 | 中文法律场景表现 | 部署成本 |
|------|------|--------|----------------|---------|
| bge-large-zh-v1.5 | 1024 | 326M | 通用中文基线，稳定 | CPU 可跑 |
| BGE-M3 | 1024 | 568M | 多语言强但中文垂直域不一定更好 | 需 GPU |
| Qwen3-Embedding | 2560 | - | 排行榜高但维度高导致存储和检索成本翻倍 | 需 GPU + 高维向量库 |

**你要能讲**：「排行榜不可信，我用自己标注的 80 条法律测试集跑了对比。bge-large-zh Top-5 命中率 XX%，BGE-M3 是 XX%，差 X 个点，但 M3 推理速度慢 X 倍。在我的场景下选了 bge-large + rerank，因为 rerank 能补回来 X 个点，综合成本更低。」

### 对比 3：向量后端
| 方案 | 特点 | 适用 |
|------|------|------|
| Chroma | 嵌入式，零运维 | 本地开发/千级文档 |
| Qdrant | Rust 实现，性能好，支持过滤 | 万级文档生产 |
| Milvus | 分布式，十亿级 | 企业大规模 |
| pgvector | 复用 PG 生态 | 已有 PG 的团队，但有 2000 维上限 |

**你要能讲**：「开发阶段用 Chroma 因为零配置，代码里抽象了 vector_index 门面层，切 Qdrant 只需改一个配置。选 Chroma 做默认是因为我的法律库 1-3K 文档，Chroma 够用；如果到 10 万级我会切 Qdrant，因为它支持载荷过滤可以在检索时直接按分库筛选。」

### 对比 4：Agent 架构
| 方案 | 特点 | 适用 |
|------|------|------|
| 直线流（你现在的） | 固定路径 policy→retrieve→draft | 简单 QA，**不是 Agent** |
| ReAct | LLM 自主决定：思考→选工具→观察结果→循环 | **生产级 Agent** |
| Plan-and-Execute | 先整体规划再分步执行 | 复杂多步任务 |
| CoT | 一次性思维链，不调工具 | 纯推理场景 |

**你要能讲**：「我一开始做的是直线流，面试官一看 Trace 就知道不是 Agent。改成 ReAct 后，LLM 自己判断需不需要查法条库、需不需要查案例库、还是直接回答。工具定义用 JSON Schema，LLM 填参数，我加了三层保护：最大 5 次迭代防死循环、连续重复检测、异常自动回退到直接回答。」

### 对比 5：记忆方案
| 方案 | token 消耗 | 信息完整性 | 实现复杂度 |
|------|-----------|-----------|----------|
| 全量历史 | 爆炸 | 100% | 无 |
| 滑动窗口（保留最近 N 轮） | 线性 | 丢远期信息 | 低 |
| 摘要压缩（每轮生成摘要） | 恒定 | 摘要质量依赖 LLM | 中 |
| **四级分层折叠**（腾讯云方案） | 按需展开 | 可恢复 | 高 |

**你要能讲**：「滑动窗口简单但会丢信息，用户第 1 轮说的诉讼时效到第 8 轮就没了。摘要压缩好一些但摘要质量不稳定。我用了四级分层：原始证据卸载到文件只留路径引用，工具调用生成 JSONL 摘要，任务进度用 Mermaid 画布，紧急时只保留元信息。消融实验显示 token 节省 XX%，任务通过率从 XX% 提升到 XX%。」

## 四、面试讲法框架

### 项目一句话介绍（15秒版）
「我做了一个法律领域的 RAG + Agent 系统，核心解决三个问题：口语化提问和专业术语的语义鸿沟、多类型法律文档（法条/司法解释/案例/问答）的精准路由、以及长对话场景下 Agent 上下文膨胀导致的 token 浪费和任务失败。」

### 简历描述（STAR-L 格式）
**S**：法律知识库检索中，口语化表述（"欠钱不还多久过期"）与专业术语（"诉讼时效"）的语义鸿沟严重，纯向量检索 Top-1 命中率不足 50%
**T**：设计并实现自适应 RAG + ReAct Agent 系统
**A**：
- 采用 BM25+向量双路检索 + RRF 融合，替代 max() 后 Top-1 提升 XX pp
- 实现三阶段域路由（规则+embedding+LLM），四分库路由准确率 XX%
- 引入 ReAct 工具路由替代直线流，工具选择准确率 XX%
- 实现四级上下文分层压缩，token 节省 XX%，长对话任务通过率提升 XX pp
- 建立全模块评估体系（检索/路由/护栏/Agent/端到端五层指标）
**R**：Top-1 命中率从 XX% 提升至 XX%，幻觉率降至 XX%，P99 延迟 XX ms
**L**：分库召回比换 Embedding 模型性价比高 10 倍；RRF 的鲁棒性优于固定权重

### 5 个必备追问逐字稿

**Q1: 为什么选法律域？**
> 法律文档天然分四类——法条、司法解释、案例、问答。这让我能验证分库路由的效果，而不是所有文档混在一起。而且法律文本有严格的条款编号（如"民法典第1062条"），能同时检验 BM25 精确匹配和向量语义检索的互补效果。

**Q2: RRF 的 k 参数怎么定的？**
> k 默认 60，是 RRF 原论文的推荐值。我在法律测试集上用 k=20/40/60/80 跑了网格搜索，60 在 Top-1 和 Top-5 上综合最优。k 太小会过度放大排名靠前的结果，k 太大会让排名差异被抹平。

**Q3: 你的 Agent 工具调用失败了怎么办？**
> 三层保护。第一层是最大迭代次数 = 5，超过直接用已有结果生成回答。第二层是循环检测，如果连续两次调用同一个工具且参数相同，判定为死循环并跳出。第三层是异常回退，工具调用抛异常时降级为不调工具直接回答，并在 audit_trace 里记录失败原因。

**Q4: 记忆压缩会不会丢信息？**
> 关键设计是可恢复。原文存在 refs 目录，摘要存在 JSONL，画布存在 MMD 文件，上下文只留元信息和路径引用。任何一层信息都能通过路径找回。摘要质量是风险点，所以我保留了原始证据的完整路径做兜底。消融实验里压缩后任务通过率不降反升，因为去掉了噪声信息后 LLM 反而更聚焦。

**Q5: 你的评估体系是什么？**
> 不只是 RAG 有数字。检索层看 Top-1/Top-5 命中率和域 Top-1；路由层看域准确率和 fallback 率；护栏层看拦截率和误拦率；Agent 层看任务完成率和工具选择准确率；端到端看 P99 延迟和 token 消耗。每次改动都跑回归，用 80+ 条标注测试集覆盖 5 类坏例。

## 六、核心认知（就业文档精华提炼）

1. **面试官看的不是你用了什么技术，是你能不能讲清楚你在系统里做了什么真实的决策**
2. **有基线有结果，面试官才信** — 没有量化指标的项目描述，默认打问号
3. **分库召回比换 Embedding 模型管用 10 倍** — 优先级：数据清洗 > 分块策略 > 召回策略 > Embedding 模型
4. **Agent 的分水岭是自主决策** — 固定流程是 Chatbot，动态工具选择才是 Agent
5. **记忆是 Agent 智能化的底座** — 没有记忆的 Agent 每次都像第一次见面
6. **真实感比完美答案值钱** — 答不上来说"我不确定"，比胡编强 100 倍
7. **每个数字背后要有故事** — "Top-1 从 64% 提到 82%" 不够，要说"因为口语化查询在 BM25 路命中率很低，RRF 让向量路的语义匹配能补回来"
