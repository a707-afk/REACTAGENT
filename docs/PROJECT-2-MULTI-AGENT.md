## 第一部分：Agent 从 0 到 1 扫盲（你说不懂，这里讲透）

### 1.1 到底什么是 Agent？

先破除误解。很多人以为"调用了 LLM + 做了几步 = Agent"，**错**。

```
Chatbot：    用户问 → LLM 答                         （无状态、无工具、无记忆）
工作流/Pipeline：用户问 → A → B → C → D → 答          （有工具，但路径写死）
Agent：      用户问 → LLM 自己决定下一步做什么 → 循环   （LLM 是决策者，路径动态）
```

**核心分水岭：谁决定下一步做什么？**
- 路径写死在代码里（if-else / 固定顺序）→ 是 Pipeline
- LLM 在运行时自己决定调哪个工具、要不要继续、何时结束 → 才是 Agent

> 面试致命题："RAG + Chat 算 Agent 吗？" 答案：不算。它是固定流水线。这题答错直接出局。

### 1.2 Agent 的四个核心组件

```
        ┌─────────────────────────────────────┐
        │              Agent                   │
        │                                      │
        │   ┌──────────┐      ┌──────────┐    │
        │   │ Planning │◀────▶│  Memory  │    │
        │   │ 规划     │      │  记忆     │    │
        │   └────┬─────┘      └────▲─────┘    │
        │        │                 │          │
        │        ▼                 │          │
        │   ┌──────────┐      ┌────┴─────┐    │
        │   │  Action  │─────▶│  Tools   │    │
        │   │  行动     │      │  工具     │    │
        │   └──────────┘      └──────────┘    │
        └─────────────────────────────────────┘
```

| 组件 | 作用 | 关键技术 |
|------|------|---------|
| **Planning（规划）** | 把大任务拆成可执行的小步骤 | 任务分解、ReAct、Plan-and-Execute、反思 |
| **Memory（记忆）** | 记住对话历史、中间结果、长期知识 | 短期记忆、长期记忆、上下文压缩 |
| **Tools（工具）** | 让 LLM 能调用外部能力（搜索、计算、API） | Function Calling、工具描述、参数 schema |
| **Action（行动）** | 实际执行工具调用并观察结果 | 工具执行、结果解析、错误处理 |

### 1.3 Agent 范式演进（从简到繁，面试要会讲）

| 范式 | 一句话原理 | 适合场景 | 缺点 |
|------|-----------|---------|------|
| **Function Calling** | LLM 直接输出要调用的函数和参数 | 简单工具调用 | 单步，无规划 |
| **ReAct** | Thought→Action→Observation 循环（边想边做） | 通用任务 | 可能跑偏/死循环 |
| **Plan-and-Execute** | 先规划完整步骤，再逐步执行 | 复杂多步任务 | 计划错了代价大 |
| **Reflection（反思）** | 生成后自我评估、改进 | 质量要求高 | 多次调用费 token |
| **Reflexion** | 反思 + 结构化记忆迭代 | 需要试错学习 | 实现复杂 |
| **Multi-Agent（多智能体）** | 多个专职 Agent 协作 | 大型复杂任务 | 协调开销大 |

**ReAct 的循环长这样**（最重要，必须能默写）：
```
Thought: 我需要先了解这个公司的基本情况
Action: web_search("XX公司 2025 财报")
Observation: [搜索结果...]
Thought: 财报显示营收增长，但我需要对比竞品
Action: web_search("XX竞品 2025 营收")
Observation: [搜索结果...]
Thought: 信息足够了，可以总结
Action: finish(总结报告)
```

### 1.4 为什么需要多 Agent？（项目二的立项理由）

单个 Agent 做复杂研究有三个问题：
1. **上下文爆炸**：一个 Agent 又要搜索、又要分析、又要写作，上下文很快塞满
2. **能力不专**：写代码和写文案需要不同的 prompt 和工具，混在一起效果差
3. **难以并行**：串行做完所有步骤太慢

多 Agent 把任务分给专职 Agent，各管一摊，像一个研究团队：
```
   主管 Agent（Orchestrator）
        │ 分派任务
   ┌────┼────┬──────────┐
   ▼    ▼    ▼          ▼
检索   分析  批判       写作
Agent  Agent Agent     Agent
（各有自己的工具、prompt、记忆）
```

## 第二部分：项目完整方案

### 2.0 项目定位

**深度研究系统（Deep Research Agent）**：用户输入一个研究问题（如"分析2025年国产大模型的商业化路径"），系统：
1. 自动拆解为子问题（规划）
2. 多 Agent 并行检索、分析（工具 + 多智能体）
3. 交叉验证、批判性审查（反思）
4. 产出结构化研究报告 + 引用溯源（生成）

这是 2026 最火的 Agent 形态（OpenAI Deep Research、Manus 都是这个方向），面试官一听就懂价值。

### 2.3 核心模块详解

#### 模块1：Orchestrator（主管 Agent）— 任务规划

```python
def planner_node(state):
    research_question = state["question"]
    # LLM 规划：把研究课题拆成子任务 DAG（有依赖关系）
    plan = llm.plan(research_question, schema=TaskDAG)
    # 示例输出：
    # {
    #   "subtasks": [
    #     {"id":"t1","goal":"收集国产大模型厂商列表","agent":"retriever","deps":[]},
    #     {"id":"t2","goal":"分析各厂商商业模式","agent":"analyst","deps":["t1"]},
    #     {"id":"t3","goal":"对比定价策略","agent":"analyst","deps":["t1"]},
    #     {"id":"t4","goal":"批判性审查结论","agent":"critic","deps":["t2","t3"]},
    #     {"id":"t5","goal":"撰写研究报告","agent":"writer","deps":["t4"]}
    #   ]
    # }
    return {"plan": plan, "task_queue": topological_sort(plan)}
```

**面试讲法**：
- "为什么用 DAG 不用线性步骤？" → 有些子任务无依赖可并行（t2、t3），DAG 能表达并行和依赖，提升效率
- "规划错了怎么办？" → Plan-and-Execute 的弱点。我加了重规划机制：执行中发现计划不合理，Orchestrator 可以动态调整后续任务

#### 模块2：子 Agent（专职 ReAct Agent）

每个子 Agent 是一个独立的 ReAct 循环，有自己的工具集和 prompt：

```python
class ReActAgent:
    def __init__(self, role, tools, system_prompt):
        self.role = role          # retriever / analyst / critic / writer
        self.tools = tools        # 该角色能用的工具子集
        self.system_prompt = system_prompt

    def run(self, task, memory):
        for step in range(MAX_STEPS):
            # Think：LLM 决定下一步
            thought, action = self.llm_decide(task, memory, self.tools)
            if action.type == "finish":
                return action.result
            # 循环检测
            if self.is_loop(action):
                return self.fallback()
            # Act：执行工具（带超时、重试、异常捕获）
            try:
                observation = self.execute_tool(action, timeout=30)
            except ToolError as e:
                observation = f"工具失败：{e}，尝试其他方法"
            # 写入工作记忆
            memory.append(thought, action, observation)
        return self.summarize(memory)  # 达到最大步数，强制总结
```

**工具定义示例（JSON Schema）**：
```json
{
  "name": "web_search",
  "description": "联网搜索最新信息。当需要时效性数据或知识库没有的信息时使用",
  "parameters": {
    "query": {"type": "string", "description": "搜索关键词"},
    "recency": {"type": "string", "enum": ["day","week","month","year"]}
  }
}
```

**面试核心**：
- "工具描述为什么重要？" → description 质量直接决定 LLM 选对工具的准确率，这是工具选择准确率指标的关键
- "工具调用失败怎么处理？" → 三层：超时控制、自动重试（指数退避）、异常降级（告诉 LLM 失败原因让它换方法）

#### 模块3：记忆系统（Agent 智能化的底座）

这是面试高频深挖点。分层设计：

| 记忆层 | 内容 | 存储 | 生命周期 |
|--------|------|------|---------|
| 工作记忆 | 当前子任务的 think-act-observe | Prompt 内 | 子任务结束清空 |
| 短期记忆 | 当前研究会话的关键中间结果 | Redis | 会话结束 |
| 长期记忆 | 跨会话的知识、用户偏好 | Qdrant 向量库 | 永久 |
| 共享黑板 | 多 Agent 间共享的任务状态和产出 | PostgreSQL | 任务结束归档 |

**上下文压缩——四级分层折叠**（长研究任务必需，否则上下文爆炸）：
```
Level 1（原文）：检索到的完整文档 → 卸载到文件，上下文只留路径引用
Level 2（摘要）：每次工具调用结果 → 生成 JSONL 摘要
Level 3（进度）：任务进度 → Mermaid 画布/结构化进度表
Level 4（元信息）：紧急压缩时只保留任务目标和关键结论
```

**面试讲法（带消融实验数据）**：
- "为什么不全量保留上下文？" → 深度研究可能调用几十次工具，全量保留 token 线性爆炸，且 LLM 注意力会被噪声分散
- "压缩会丢信息吗？" → 关键设计是可恢复：原文存文件留路径，摘要是索引不是替代，需要细节时按路径取回
- "效果数据？" → 消融实验：有记忆压缩 vs 无压缩，token 节省 XX%，长任务完成率从 XX% 提升到 XX%（这是硬数据）

#### 模块4：Critic Agent（批判性审查）— 质量门控

```python
def critic_node(state):
    findings = state["findings"]  # 各 Agent 的产出
    # 批判性审查：逻辑漏洞、证据不足、自相矛盾、缺少反面观点
    review = critic_llm.review(findings, criteria=[
        "结论是否有充分证据支撑",
        "是否存在自相矛盾",
        "是否遗漏重要反面观点",
        "数据来源是否可靠"
    ])
    if review.has_issues:
        # 打回相关子任务重做（带具体反馈）
        return {"action": "rework", "feedback": review.issues}
    return {"action": "proceed"}
```

**面试讲法**：这就是 Reflection（反思）模式的应用。生成-评估-改进闭环，是 Agent 自我优化能力的体现。

### 2.4 端到端数据流（一次深度研究）

```
用户："分析 2025 年国产大模型的商业化路径"
    ↓
[Orchestrator-规划] 拆解为 5 个子任务的 DAG（t1→t2,t3→t4→t5）
    ↓
[调度] t1 无依赖先执行 → Retriever Agent
    [Retriever] ReAct循环：web_search厂商列表 → rag_retrieve行业报告 → 整理出厂商清单
    ↓
[调度] t1完成，t2/t3无相互依赖 → 并行执行
    [Analyst-1] 分析商业模式（ReAct：检索+计算+推理）
    [Analyst-2] 对比定价策略（并行）
    ↓
[调度] t2/t3完成 → t4 批判审查
    [Critic] 发现"定价对比缺少 To B/To C 区分" → 打回 t3 重做
    [Analyst-2] 带反馈重做 → 补充 To B/To C 维度
    ↓
[调度] t4通过 → t5 写作
    [Writer] 整合所有发现 → 流式生成结构化报告 + 引用溯源
    ↓
[全程] SSE 实时推送每个 Agent 的 think/act/observe 到前端；
       Langfuse 记录完整决策树；记忆系统分层压缩控制 token
    ↓
[返回] 带引用的研究报告 + 可视化的 Agent 协作过程
```

### 2.7 面试讲法（STAR-L）

**S**：单 Agent 处理复杂研究任务时，上下文爆炸、能力不专、无法并行，导致长任务完成率低、token 成本高。

**T**：设计一个多 Agent 深度研究系统，支持自主规划、工具调用、多智能体协作、长期记忆。

**A**：
- Orchestrator 把研究课题规划为子任务 DAG，支持并行和依赖调度
- 实现专职 ReAct Agent（检索/分析/批判/写作），各有工具集和 prompt
- 工具层用 JSON Schema 定义 + 沙箱隔离 + 超时重试降级三层保护
- 四层记忆 + 四级上下文压缩，长任务 token 节省 XX%
- Critic Agent 反思闭环做质量门控，不合格自动重做
- Langfuse 追踪多 Agent 决策树，SSE 实时展示思考过程

**R**：端到端任务成功率 XX%，相比单 Agent 提升 XX pp，token 节省 XX%，并行加速比 XX。

**L**：
- 记忆压缩是长任务的命门，没有它 Agent 跑几步就上下文溢出
- 多 Agent 不是越多越好，协调开销和收益要权衡
- Critic 审查显著降低了幻觉和逻辑漏洞

### 必背追问
1. "为什么用多 Agent 不用单 Agent？" → 上下文隔离、能力专精、可并行；但有协调开销，要看任务复杂度
2. "ReAct 死循环怎么办？" → 最大步数 + 连续重复检测 + 强制总结降级
3. "多 Agent 怎么通信？" → 共享黑板（PostgreSQL）+ Orchestrator 调度，不是 Agent 直接互相喊话
4. "记忆怎么压缩还不丢信息？" → 四级分层，原文卸载留路径，可恢复
5. "工具选择不准怎么办？" → 优化工具 description + few-shot 示例 + 工具选择准确率监控
6. "和 LangChain Agent 有什么区别？" → LangGraph 的图状态机对循环、分支、人在回路的控制力更强，多 Agent 状态更可控
7. "成本怎么控制？" → 分级用模型（强模型规划，弱模型执行简单子任务）+ 记忆压缩 + 缓存

## 第三部分：两个项目的关系与简历组合

### 互补性
| 维度 | 项目一（Agentic RAG） | 项目二（Multi-Agent） |
|------|---------------------|---------------------|
| 主打能力 | RAG 工程深度 | Agent 研发深度 |
| 核心卖点 | 混合检索/路由/幻觉治理/评估 | 规划/工具/记忆/多智能体 |
| 对应岗位 | RAG 工程师、知识库工程师 | Agent 研发、应用开发 |
| 复用关系 | 项目二的 rag_retrieve 工具复用项目一 | 两项目打通，体现系统设计能力 |

### 简历呈现建议
- 两个项目放一起，第一个证明你 RAG 扎实，第二个证明你 Agent 能落地
- 强调项目二的 `rag_retrieve` 工具直接复用项目一 → 体现你能做系统集成，不是孤立的玩具
- 每个项目 4-6 条 bullet，每条都带数字

### 学习顺序建议
1. **先做项目一**（你已有基础，升级改造）→ 把 RAG 吃透，建立评估习惯
2. **再做项目二**（全新，但能复用项目一的检索作为工具）→ 攻克 Agent
3. 两个都做完，AB 两条线（RAG 工程 + Agent 研发）都覆盖了
